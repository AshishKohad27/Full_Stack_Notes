# 36. OOPs Complete Guide - From Zero to Interview Ready

## What is OOP?
- OOP = **Object Oriented Programming**
- Organize code around **objects** (real world things) instead of just functions
- 4 Pillars: **Encapsulation, Inheritance, Polymorphism, Abstraction**

---

## PILLAR 1: ENCAPSULATION (Data Hiding)

### What?
- Wrap data (variables) and methods together in a class
- Hide internal details, expose only what's needed

### Why?
- Protection: nobody can mess with your data directly
- Control: you decide what others can see/modify

### Example
```java
class ATM {
    private double balance; // HIDDEN from outside

    ATM(double balance) {
        this.balance = balance;
    }

    // Only these methods can touch balance
    public void deposit(double amount) {
        if (amount > 0) {
            balance += amount;
            System.out.println("Deposited: " + amount);
        } else {
            System.out.println("Invalid amount!");
        }
    }

    public void withdraw(double amount) {
        if (amount > 0 && amount <= balance) {
            balance -= amount;
            System.out.println("Withdrawn: " + amount);
        } else {
            System.out.println("Insufficient balance or invalid amount!");
        }
    }

    public double getBalance() {
        return balance; // read-only access
    }
}

class Hello {
    public static void main(String args[]) {
        ATM atm = new ATM(10000);

        // atm.balance = -99999; // ERROR! private - can't access directly
        // If balance was public, anyone could set it to negative!

        atm.deposit(5000);        // Deposited: 5000
        atm.withdraw(3000);       // Withdrawn: 3000
        System.out.println("Balance: " + atm.getBalance()); // 12000
    }
}
```

### Interview Trick: Encapsulation vs Abstraction
```
Encapsulation = HOW you hide (using private + getters/setters)
Abstraction   = WHAT you hide (implementation details)

Encapsulation is the mechanism, Abstraction is the concept.
Example: ATM machine
  - Abstraction: you press "Withdraw", you don't know the internal process
  - Encapsulation: the code inside ATM is private, you use public buttons
```

## PILLAR 2: INHERITANCE (Code Reuse)

### What?
- Child class **gets** all properties and methods of parent class
- "IS-A" relationship: Dog IS-A Animal

### Why?
- Don't repeat code - write once in parent, all children get it

### Types
```
1. Single:       Parent --> Child
2. Multilevel:   Grandparent --> Parent --> Child
3. Hierarchical: Parent --> Child1, Parent --> Child2

NOT ALLOWED in Java:
4. Multiple:     Parent1 --> Child <-- Parent2 (use Interfaces instead)
```

### Example
```java
class Phone {
    String brand;
    int battery;

    Phone(String brand, int battery) {
        this.brand = brand;
        this.battery = battery;
    }

    void call(String number) {
        System.out.println(brand + " calling " + number + "...");
    }

    void showBattery() {
        System.out.println("Battery: " + battery + "%");
    }
}

// SmartPhone IS-A Phone + extra features
class SmartPhone extends Phone {
    int storage;

    SmartPhone(String brand, int battery, int storage) {
        super(brand, battery); // call parent constructor
        this.storage = storage;
    }

    // SmartPhone's own method
    void takePhoto() {
        System.out.println(brand + " taking photo...");
    }

    // Override parent method
    @Override
    void call(String number) {
        System.out.println(brand + " video calling " + number + "...");
    }
}

class Hello {
    public static void main(String args[]) {
        SmartPhone sp = new SmartPhone("iPhone", 85, 128);
        sp.call("9876543210");  // iPhone video calling 9876543210...
        sp.showBattery();        // Battery: 85% (inherited)
        sp.takePhoto();          // iPhone taking photo... (own method)
    }
}
```

### Interview Trick: What is NOT inherited?
```
1. Constructors - NOT inherited (but parent constructor IS called via super)
2. Private members - NOT accessible in child (they exist but are hidden)
3. Static methods - NOT overridden (they are HIDDEN)
```

### Interview Trick: Diamond Problem
```
Why Java doesn't allow multiple inheritance with classes:

class A { void show() { print("A"); } }
class B extends A { void show() { print("B"); } }
class C extends A { void show() { print("C"); } }
class D extends B, C { } // Which show()? B's or C's? AMBIGUOUS!

Solution: Java uses INTERFACES for multiple inheritance
```

## PILLAR 3: POLYMORPHISM (Many Forms)

### What?
- Same thing behaves differently in different situations
- 2 types: **Compile-time** (overloading) + **Runtime** (overriding)

### Compile-time Polymorphism (Method Overloading)
```java
class Printer {
    // Same method name, different parameters
    void print(String s)        { System.out.println("String: " + s); }
    void print(int n)           { System.out.println("Int: " + n); }
    void print(String s, int n) { System.out.println(s + " x" + n); }
}

class Hello {
    public static void main(String args[]) {
        Printer p = new Printer();
        p.print("Hello");      // String: Hello
        p.print(42);           // Int: 42
        p.print("Hello", 3);   // Hello x3
        // Java decides WHICH method at COMPILE time based on arguments
    }
}
```

### Runtime Polymorphism (Method Overriding) -- INTERVIEW FAVORITE!
```java
class Shape {
    void area() {
        System.out.println("Calculating area...");
    }
}

class Circle extends Shape {
    double radius;
    Circle(double r) { radius = r; }

    @Override
    void area() {
        System.out.println("Circle area: " + (3.14 * radius * radius));
    }
}

class Rectangle extends Shape {
    double length, width;
    Rectangle(double l, double w) { length = l; width = w; }

    @Override
    void area() {
        System.out.println("Rectangle area: " + (length * width));
    }
}

class Hello {
    public static void main(String args[]) {

        // PARENT reference, CHILD object -- KEY to runtime polymorphism
        Shape s1 = new Circle(5);
        Shape s2 = new Rectangle(4, 6);

        s1.area(); // Circle area: 78.5
        s2.area(); // Rectangle area: 24.0

        // Java decides WHICH area() to call at RUNTIME based on actual object
        // Reference type = Shape, but actual object decides the behavior

        // Array of different shapes - all treated as Shape
        Shape shapes[] = { new Circle(3), new Rectangle(2, 5), new Circle(7) };
        for (Shape s : shapes) {
            s.area(); // each calls its OWN area()
        }
    }
}
```

### Interview Trick: Overloading vs Overriding

| Feature        | Overloading               | Overriding                 |
|----------------|---------------------------|----------------------------|
| Where          | Same class                | Parent + Child class       |
| Method name    | Same                      | Same                       |
| Parameters     | MUST be different         | MUST be same               |
| Return type    | Can be different          | Must be same (or covariant)|
| Access         | Can be any                | Same or wider              |
| static?        | Can overload static       | Cannot override static     |
| final?         | Can overload final        | Cannot override final      |
| Binding        | Compile-time (early)      | Runtime (late/dynamic)     |

### Interview Trick: Can you Override...?
```
static method   --> NO (it's method hiding, not overriding)
final method    --> NO (compilation error)
private method  --> NO (not visible to child, it's a new method)
constructor     --> NO (constructors are not inherited)
main() method   --> YES (but both parent and child main exist independently)
```

## PILLAR 4: ABSTRACTION (Hide Complexity)

### What?
- Show only **what** something does, hide **how** it does it
- Achieved using: **Abstract Classes** and **Interfaces**

### Abstract Class (0% to 100% abstraction)
```java
abstract class Vehicle {
    String brand;

    Vehicle(String brand) {
        this.brand = brand;
    }

    // Abstract method - child MUST implement
    abstract void start();

    // Normal method - child gets for free
    void stop() {
        System.out.println(brand + " stopped");
    }
}

class ElectricCar extends Vehicle {
    ElectricCar(String brand) { super(brand); }

    @Override
    void start() {
        System.out.println(brand + " starts silently (electric)");
    }
}

class PetrolCar extends Vehicle {
    PetrolCar(String brand) { super(brand); }

    @Override
    void start() {
        System.out.println(brand + " starts with engine roar (petrol)");
    }
}

class Hello {
    public static void main(String args[]) {
        Vehicle v1 = new ElectricCar("Tesla");
        Vehicle v2 = new PetrolCar("BMW");

        v1.start(); // Tesla starts silently (electric)
        v2.start(); // BMW starts with engine roar (petrol)
        v1.stop();  // Tesla stopped
    }
}
```

### Interface (100% abstraction)
```java
interface Flyable {
    void fly(); // abstract by default
}

interface Swimmable {
    void swim();
}

// Class can implement MULTIPLE interfaces (solves diamond problem!)
class Duck implements Flyable, Swimmable {
    public void fly() {
        System.out.println("Duck flies");
    }
    public void swim() {
        System.out.println("Duck swims");
    }
}

class Hello {
    public static void main(String args[]) {
        Duck d = new Duck();
        d.fly();  // Duck flies
        d.swim(); // Duck swims

        // Interface as reference type
        Flyable f = new Duck();
        f.fly(); // Duck flies
        // f.swim(); // ERROR! Flyable doesn't know about swim()
    }
}
```

### Interview Trick: Abstract Class vs Interface

| Feature              | Abstract Class           | Interface                  |
|----------------------|--------------------------|----------------------------|
| Methods              | Abstract + Normal        | All abstract (before Java 8)|
| Variables            | Any type                 | public static final only   |
| Constructor          | Yes                      | No                         |
| Multiple inheritance | No (single extends)      | Yes (implement many)       |
| Access modifiers     | Any                      | public only                |
| When to use?         | "IS-A" with shared code  | "CAN-DO" capability        |
| Speed                | Fast                     | Slightly slower            |

### Java 8 Interface Update: default and static methods
```java
interface Greeting {
    void sayHi(); // abstract

    // Default method - has body! (Java 8+)
    default void sayBye() {
        System.out.println("Goodbye!");
    }

    // Static method in interface (Java 8+)
    static void info() {
        System.out.println("Greeting interface v1.0");
    }
}

class English implements Greeting {
    public void sayHi() {
        System.out.println("Hello!");
    }
    // sayBye() is already implemented - optional to override
}

class Hello {
    public static void main(String args[]) {
        English e = new English();
        e.sayHi();    // Hello!
        e.sayBye();   // Goodbye! (default method)
        Greeting.info(); // Greeting interface v1.0 (static)
    }
}
```

## BONUS: OOP Interview Questions with Answers

### Q1: What is the difference between class and object?
```
Class  = Blueprint (no memory, logical)
Object = Instance of class (has memory, physical)
You can create multiple objects from one class
```

### Q2: What is IS-A and HAS-A relationship?
```java
// IS-A = Inheritance
class Dog extends Animal { } // Dog IS-A Animal

// HAS-A = Composition (object inside object)
class Car {
    Engine engine; // Car HAS-A Engine (not IS-A)
}
class Engine {
    int horsepower;
}
```

### Q3: Why is multiple inheritance not allowed?
```
Diamond Problem: If class C extends both A and B,
and both A and B have method show(), which one does C get?
Ambiguous! Java avoids this by using interfaces instead.
Interfaces solve it because the implementing class provides its own body.
```

### Q4: Can an abstract class have a constructor?
```
YES! It's called when child class object is created (via super())
You just can't do: new AbstractClass() directly
```

### Q5: Can we create object of an interface?
```
No! But you can use interface as reference type:
  Flyable f = new Duck(); // Duck implements Flyable
This is polymorphism - Flyable reference, Duck object
```

### Q6: What is marker interface?
```
An interface with NO methods. It just "marks" a class.
Example: Serializable, Cloneable
class Student implements Serializable { } // marks Student as serializable
```

### Q7: What is covariant return type?
```java
class Parent {
    Parent get() { return this; }
}
class Child extends Parent {
    @Override
    Child get() { return this; } // return type is Child, not Parent -- ALLOWED!
}
// Covariant = child override can return child type instead of parent type
```

### Q8: Association, Aggregation, Composition
```
Association: two objects are related (Teacher teaches Student)
Aggregation: "HAS-A" - weak (Department has Professors, profs can exist without dept)
Composition: "HAS-A" - strong (House has Rooms, rooms can't exist without house)
```
