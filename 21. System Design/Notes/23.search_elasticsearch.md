# 23. Search & Elasticsearch

- Normal database search (SQL LIKE) is **SLOW** for text search
- Elasticsearch = a **search engine** built for fast full-text search
- Used by: Google, Wikipedia, GitHub, Amazon product search

## Problem: Why Not Just Use SQL?

```
SQL Search:
  SELECT * FROM products WHERE name LIKE '%wireless headphone%';

Problems:
  1. LIKE '%keyword%' → does FULL TABLE SCAN → SLOW on millions of rows
  2. Can't handle typos: "wireles headphon" → NO results
  3. Can't rank by relevance
  4. Can't do fuzzy matching

Elasticsearch:
  - Searches millions of documents in milliseconds
  - Handles typos: "wireles" → still finds "wireless"
  - Ranks results by relevance
  - Supports autocomplete, suggestions
```

## How Elasticsearch Works (Inverted Index)

```
Normal Index (Database):
  Document 1 → "The quick brown fox"
  Document 2 → "The quick blue car"
  Document 3 → "A brown dog"

Inverted Index (Elasticsearch):
  "the"   → [Doc 1, Doc 2]
  "quick" → [Doc 1, Doc 2]
  "brown" → [Doc 1, Doc 3]
  "fox"   → [Doc 1]
  "blue"  → [Doc 2]
  "car"   → [Doc 2]
  "dog"   → [Doc 3]

Search "brown" → instantly returns [Doc 1, Doc 3] → FAST!
Like the INDEX at the back of a textbook!
```

## Example: E-commerce Search

```
User searches: "blue running shoes"

Elasticsearch:
  1. Splits into: ["blue", "running", "shoes"]
  2. Looks up inverted index for each word
  3. Finds documents matching ALL or SOME words
  4. Ranks by relevance (all 3 words match > 2 words match)
  5. Returns results in milliseconds

Results (ranked):
  1. "Blue Running Shoes Nike" (score: 10 - all words match)
  2. "Blue Casual Shoes" (score: 7 - 2 words match)
  3. "Red Running Shoes" (score: 6 - 2 words match)
```

## Architecture in System Design

```
User searches → API Gateway → Search Service → Elasticsearch
                                              → returns results

Data sync (keep ES in sync with database):
  Database (source of truth) → Message Queue → Elasticsearch

When product is added/updated in DB:
  1. Save to Database (MySQL)
  2. Send event to Message Queue (Kafka)
  3. Consumer updates Elasticsearch

This way: DB is source of truth, ES is for searching
```

```
User types "iphon" → Autocomplete Service → Elasticsearch
                                           → suggestions: ["iphone 15", "iphone case", "iphone charger"]
```

## When to Use Elasticsearch?

```
USE Elasticsearch:
  ✅ Full-text search (search across text content)
  ✅ Autocomplete / suggestions
  ✅ Log analysis (search through millions of logs)
  ✅ Product search (e-commerce)
  ✅ Fuzzy search (handle typos)

DON'T USE for:
  ❌ Primary database (not ACID compliant)
  ❌ Transactions (banking, payments)
  ❌ Frequently updated data (ES is optimized for reads)
```

## Key Points
- Elasticsearch uses **inverted index** for fast text search
- Much faster than SQL `LIKE` queries for text search
- Handles **typos, synonyms, relevance ranking**
- Keep ES **in sync** with your database via message queue
- ES is for **searching**, NOT a replacement for your database
- Used for: product search, log analysis, autocomplete
