## Component Quick Reference

| Component        | What it Does                          | Example Tools              |
|------------------|---------------------------------------|----------------------------|
| Load Balancer    | Distributes traffic across servers    | Nginx, HAProxy, AWS ELB    |
| Cache            | Fast temporary storage                | Redis, Memcached           |
| CDN              | Serve static content globally         | Cloudflare, CloudFront     |
| Message Queue    | Async communication between services  | Kafka, RabbitMQ, SQS       |
| API Gateway      | Single entry point, auth, routing     | Kong, Nginx, AWS API GW    |
| Database (SQL)   | Structured data, relationships        | PostgreSQL, MySQL          |
| Database (NoSQL) | Flexible, scalable data               | MongoDB, Cassandra, Redis  |
| Object Storage   | Store files, images, videos           | AWS S3, Google Cloud Store |
| Search Engine    | Full-text search                      | Elasticsearch, Solr        |
| DNS              | Domain name to IP resolution          | Route53, Cloudflare DNS    |

## Scaling Cheat Sheet

```
Problem                          → Solution
──────────────────────────────────────────────
Too much traffic on server       → Load Balancer + more servers
Database reads are slow          → Read replicas + Cache (Redis)
Database writes are slow         → Sharding + Message Queue
Static files loading slowly      → CDN
Single point of failure          → Redundancy at every layer
Need real-time updates           → WebSocket / SSE
Background tasks blocking users  → Message Queue (async)
Too many API calls               → Rate Limiting
Global users, high latency       → Multi-region deployment + CDN
Search is slow                   → Elasticsearch + Index
```

## Numbers Every Developer Should Know

```
Read from L1 cache:           0.5 ns
Read from L2 cache:           7 ns
Read from RAM:                100 ns
Read from SSD:                150,000 ns (150 us)
Read from HDD:                10,000,000 ns (10 ms)
Network round trip (same DC):  500,000 ns (0.5 ms)
Network round trip (US to EU): 150,000,000 ns (150 ms)

Key takeaway:
  Memory (RAM/Cache) is 1000x faster than Disk
  Same datacenter is 300x faster than cross-continent
```

```

Daily Active Users (DAU): 100M
Requests per user/day: 10
Total requests/day: 1 billion
Requests per second: 1B / 86400 = ~12,000 RPS

Storage per user: 1 KB
Total storage: 100M × 1 KB = 100 GB
Storage for 5 years: 100 GB × 365 × 5 = 182 TB
```

## SQL vs NoSQL Decision

```
Choose SQL when:
  ✅ Data has relationships (users + orders + products)
  ✅ Need ACID transactions (banking, payments)
  ✅ Data structure is fixed
  ✅ Complex queries with JOINs

Choose NoSQL when:
  ✅ Data is unstructured or changes frequently
  ✅ Need horizontal scaling (petabytes of data)
  ✅ High write throughput needed
  ✅ Simple queries (key-value lookups)
```

## CAP Theorem Quick Reference

```
Can only pick 2 of 3: Consistency, Availability, Partition Tolerance

CP Systems (Consistency + Partition Tolerance):
  → MongoDB, Redis, HBase
  → Use for: Banking, inventory (need correct data)

AP Systems (Availability + Partition Tolerance):
  → Cassandra, DynamoDB, CouchDB
  → Use for: Social media, shopping cart (need uptime)
```

## Design Pattern Quick Reference

```
Pattern                  When to Use
─────────────────────────────────────
Cache-Aside             Read-heavy workloads
Write-Through Cache     Need cache consistency
Event-Driven (Pub/Sub)  Decouple services
CQRS                    Separate read/write models
Saga                    Distributed transactions across microservices
Circuit Breaker         Prevent cascade failures
Retry with Backoff      Handle temporary failures
Sharding                Database too large for one server
Leader-Follower         Database replication
```

## Interview Buzzwords to Use

```
"We can add a LOAD BALANCER to distribute traffic"

"For async processing, we can use a MESSAGE QUEUE like Kafka"

"READ REPLICAS will help handle read-heavy traffic"
"An API GATEWAY will handle auth, rate limiting, and routing"
"WebSockets for REAL-TIME communication"
"We need EVENTUAL CONSISTENCY here since strong consistency isn't required"
"Let's add REDUNDANCY to eliminate single points of failure"
```

## System Design Interview DO's and DON'Ts

```
DO:
  ✅ Ask clarifying questions first
  ✅ Start with high-level design, then go deep
  ✅ Talk about trade-offs (there's no perfect solution)
  ✅ Estimate scale (users, storage, bandwidth)
  ✅ Address bottlenecks and failures
  ✅ Use proper terminology

DON'T:
  ❌ Jump into details without understanding requirements
  ❌ Design for every edge case (focus on main flow)
  ❌ Stay silent (think out loud!)
  ❌ Over-engineer (start simple, add complexity as needed)
  ❌ Ignore scalability and failure scenarios
  ❌ Forget to mention trade-offs
```
