# 21. Authentication & Authorization

- **Authentication** = "WHO are you?" (verify identity)
- **Authorization** = "WHAT can you do?" (verify permissions)

## Real World Analogy

```
HOTEL:
  Authentication = Showing your ID at reception (proving who you are)
  Authorization  = Your room key only opens YOUR room, not others

OFFICE:
  Authentication = Swiping your ID card at entrance (proving you work here)
  Authorization  = Intern can't access server room, only admin can
```

## 1. Session-Based Authentication (Traditional)

```
Step 1: User logs in
  Client → POST /login { email: "ashish@mail.com", password: "123" }
  Server → Verifies credentials → Creates SESSION on server
  Server → Sends back Session ID as cookie

Step 2: Subsequent requests
  Client → GET /profile (sends cookie with session ID automatically)
  Server → Checks session store → "Session ABC123 belongs to Ashish"
  Server → Returns Ashish's profile

How it looks:
  Client ──cookie: session_id=ABC123──→ Server
  Server ──check session store──→ "ABC123 = Ashish, valid"
  Server ──returns data──→ Client
```

```
PROBLEM with Sessions at Scale:
  Server 1 has session → User's next request goes to Server 2
  Server 2 doesn't have the session → "Who are you??" → LOGGED OUT!

  Solutions:
  1. Sticky sessions (load balancer always sends user to same server)
  2. Shared session store (Redis) → all servers check same place
  3. Use JWT instead (stateless!)
```

## 2. JWT (JSON Web Token) -- Most Popular Today

```
Step 1: User logs in
  Client → POST /login { email: "ashish@mail.com", password: "123" }
  Server → Verifies → Creates JWT token → Sends to client

Step 2: Subsequent requests
  Client → GET /profile
           Header: Authorization: Bearer eyJhbGciOiJIUz...
  Server → Decodes JWT → Verifies signature → Returns data

NO session stored on server! Token contains all info!
```

### What's Inside a JWT?

```
JWT has 3 parts separated by dots:
  header.payload.signature

HEADER (algorithm info):
{
  "alg": "HS256",
  "typ": "JWT"
}

PAYLOAD (user data):
{
  "user_id": 123,
  "name": "Ashish",
  "role": "admin",
  "exp": 1700000000   ← expiry time
}

SIGNATURE:
  HMAC-SHA256(header + payload, SECRET_KEY)
  → Server uses this to verify token wasn't tampered with
```

### JWT Flow Example

```
1. Ashish logs in → Server creates JWT:
   { user_id: 123, role: "admin", exp: "1 hour" }
   → Signs with SECRET_KEY → sends token to Ashish

2. Ashish wants to see admin dashboard:
   → Sends JWT in header
   → Server decodes: "user_id=123, role=admin" → ALLOWED!

3. Rahul (role: "user") tries admin dashboard:
   → Sends JWT in header
   → Server decodes: "user_id=456, role=user" → DENIED! 403 Forbidden

4. Hacker modifies JWT payload to role: "admin":
   → Server verifies signature → INVALID! Signature doesn't match → DENIED!
```

## Session vs JWT

| Feature         | Session                     | JWT                          |
|-----------------|-----------------------------|-----------------------------|
| Stored where?   | Server (memory/Redis)       | Client (localStorage/cookie)|
| Stateful?       | Yes (server stores state)   | No (stateless)              |
| Scalability     | Hard (shared session store) | Easy (no server storage)    |
| Revoke access   | Easy (delete session)       | Hard (token valid till expiry)|
| Size            | Small (just session ID)     | Larger (contains user data) |

## 3. OAuth 2.0 ("Login with Google/GitHub")

```
You click "Login with Google" on a website:

Step 1: Website redirects you to Google login page
Step 2: You enter your Google credentials (on Google's page, NOT the website)
Step 3: Google says: "Allow this website to access your name and email?"
Step 4: You click "Allow"
Step 5: Google sends an Authorization Code to the website
Step 6: Website exchanges code for Access Token with Google
Step 7: Website uses Access Token to get your name/email from Google
Step 8: Website creates your account / logs you in

KEY: The website NEVER sees your Google password!
```

```
Client → Website → Google Auth Page → User Approves
                                     ↓
Website ← Authorization Code ← Google
Website → Exchange code for token → Google
Website ← Access Token ← Google
Website → GET /userinfo (with token) → Google
Website ← { name: "Ashish", email: "..." } ← Google
```

## 4. API Key Authentication

```
Simple: Include a key in every request.

GET /api/weather?api_key=abc123def456

Used for: Public APIs (Google Maps, Weather API, OpenAI)
Not for: User authentication (keys can be shared/stolen)
```

## Key Points
- **Authentication** = verify WHO (login)
- **Authorization** = verify WHAT they can do (permissions)
- **JWT** is stateless, scalable, most popular for APIs
- **OAuth** is for "Login with Google/GitHub" (third-party auth)
- **Sessions** are simpler but harder to scale
- Always use **HTTPS** (tokens sent over HTTP can be stolen!)
- Store JWT in **httpOnly cookie** (not localStorage) to prevent XSS attacks
