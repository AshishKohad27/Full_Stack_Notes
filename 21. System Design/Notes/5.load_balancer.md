# 5. Load Balancer

- Load Balancer = **traffic manager** that distributes requests across multiple servers
- Without it, one server gets ALL traffic and crashes

## How It Works

```
WITHOUT Load Balancer:              WITH Load Balancer:

  User1 ──┐                          User1 ──┐
  User2 ──┼──> Server 1 (OVERLOADED!)  User2 ──┼──> Load Balancer ──┬──> Server 1
  User3 ──┤                          User3 ──┤                   ├──> Server 2
  User4 ──┘                          User4 ──┘                   └──> Server 3

  Server handles ALL = SLOW/CRASH     Each server handles SOME = FAST
```

## Real World Analogy

```
McDonald's without Load Balancer:
  - 1 counter, 100 customers in line = SLOW

McDonald's WITH Load Balancer:
  - Manager (Load Balancer) says:
    "Customer 1 → Counter 1"
    "Customer 2 → Counter 2"
    "Customer 3 → Counter 3"
  - 3 counters, ~33 customers each = FAST
```

## Load Balancing Algorithms (How to Distribute Traffic?)

### 1. Round Robin (Most Simple)
```
Request 1 → Server 1
Request 2 → Server 2
Request 3 → Server 3
Request 4 → Server 1  (back to first, round and round)
Request 5 → Server 2
...

Simple but doesn't consider server load.
If Server 1 is slow, it still gets same traffic.
```

### 2. Weighted Round Robin
```
Server 1 (powerful)   → gets 5 requests
Server 2 (medium)     → gets 3 requests
Server 3 (small)      → gets 2 requests

Bigger server handles more traffic.
```

### 3. Least Connections
```
Send request to the server with FEWEST active connections.

Server 1: 10 connections
Server 2: 3 connections  ← next request goes here!
Server 3: 7 connections

Best for: requests that take different amounts of time.
```

### 4. IP Hash
```
Hash the client's IP address to determine which server.

Client IP 192.168.1.1 → always goes to Server 2
Client IP 192.168.1.2 → always goes to Server 1

Same client always hits same server (good for sessions).
```

### 5. Least Response Time
```
Send to the server that responds FASTEST.

Server 1: avg 200ms
Server 2: avg 50ms  ← next request goes here!
Server 3: avg 150ms
```

## Types of Load Balancers

| Type             | Works At        | What It Sees              | Example         |
|------------------|-----------------|---------------------------|-----------------|
| Layer 4 (L4)     | Transport layer | IP + Port only            | TCP/UDP routing |
| Layer 7 (L7)     | Application     | Full HTTP request (URL, headers) | Nginx, HAProxy |

```
Layer 7 is smarter:
  /api/*     → send to API servers
  /images/*  → send to image servers
  /videos/*  → send to video servers
```

## Health Checks

```
Load Balancer constantly checks: "Are you alive?"

Every 10 seconds:
  LB → Server 1: "ping!" → Server 1: "pong!" ✅ (healthy)
  LB → Server 2: "ping!" → Server 2: "pong!" ✅ (healthy)
  LB → Server 3: "ping!" → Server 3: "....."  ❌ (dead!)

LB stops sending traffic to Server 3 until it recovers.
```

## Popular Load Balancers
- **Nginx** - most popular, also works as web server
- **HAProxy** - high performance, widely used
- **AWS ELB** - Amazon's managed load balancer
- **Cloudflare** - global load balancing + CDN

## Key Points
- Load Balancer sits BETWEEN client and servers
- Distributes traffic so no single server is overwhelmed
- Removes **Single Point of Failure** (if one server dies, others handle traffic)
- Also does **health checks** to avoid sending traffic to dead servers
- Can be hardware (expensive) or software (Nginx, HAProxy)
