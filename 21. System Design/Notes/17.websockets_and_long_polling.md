# 17. WebSockets, Polling & Long Polling

- HTTP is **one-way**: client asks, server responds, connection closes
- But what about real-time apps? (chat, live scores, stock prices)
- These need the server to **push** updates to the client

## 1. Short Polling (Simple but Wasteful)

```
Client asks server every few seconds: "Any new messages?"

Client → "Any updates?" → Server: "No"     (wasted request)
(wait 3 sec)
Client → "Any updates?" → Server: "No"     (wasted request)
(wait 3 sec)
Client → "Any updates?" → Server: "Yes! New message!"
(wait 3 sec)
Client → "Any updates?" → Server: "No"     (wasted request)

Problem: Lots of empty requests = wasted bandwidth and server load
Used in: Simple dashboards that don't need instant updates
```

## 2. Long Polling (Better)

```
Client asks server, server HOLDS the connection until there's new data:

Client → "Any updates?" → Server: (holds connection... waits...)
                                   (waits...)
                                   (new data arrives!)
                          Server → "Here's the update!"
Client → "Any updates?" → Server: (holds again... waits...)

Fewer wasted requests! Server responds ONLY when there's new data.
```

```
Short Polling:  Request every 3 sec = 20 requests/min (most empty)
Long Polling:   Request held open = only responds when data exists
```

## 3. Server-Sent Events (SSE) - Server Pushes to Client

```
Client opens ONE connection → Server pushes events whenever it wants

Client ──────── open connection ──────── Server
        ←──── "New message from Rahul" ────
        ←──── "Ashish liked your post"  ────
        ←──── "New comment"             ────

ONE-WAY: Server → Client only
Client can't send data back through this connection.

Used in: Live feeds, stock tickers, notifications
```

## 4. WebSocket (Full Two-Way Communication)

```
Client and Server maintain a PERSISTENT connection.
BOTH can send messages to each other at ANY time.

Client ←─────────────────────→ Server
       → "Send message to Rahul"
       ← "Message delivered!"
       ← "Rahul is typing..."
       ← "New message from Rahul"
       → "Send: Hello!"

TWO-WAY: Both client and server can send anytime.
Connection stays open until closed.
```

### How WebSocket Works
```
Step 1: HTTP Handshake (upgrade request)
  Client → Server: "Hey, let's upgrade to WebSocket?"
  Server → Client: "Sure, upgraded!"

Step 2: Persistent Connection
  Now both can send messages freely
  No need to open/close connection each time

Step 3: Close
  Either side can close the connection
```

## Comparison

| Feature         | Short Polling    | Long Polling     | SSE              | WebSocket        |
|-----------------|------------------|------------------|------------------|------------------|
| Direction       | Client → Server  | Client → Server  | Server → Client  | Both ways        |
| Connection      | New each time    | Held open        | Persistent       | Persistent       |
| Real-time       | Near real-time   | Near real-time   | Real-time        | Real-time        |
| Overhead        | High             | Medium           | Low              | Low              |
| Complexity      | Simple           | Medium           | Simple           | Complex          |
| Best for        | Simple checks    | Notifications    | Live feeds       | Chat, gaming     |

## When to Use What?

```
Short Polling:  Weather app (updates every 30 min, no rush)
Long Polling:   Email notifications (check periodically)
SSE:            Live cricket score, stock price ticker (one-way updates)
WebSocket:      WhatsApp chat, multiplayer games, live collaboration (two-way)
```

## Real World Usage
```
WhatsApp:  WebSocket (real-time two-way chat)
Uber:      WebSocket (live driver location)
YouTube:   SSE (live view count updates)
Gmail:     Long Polling (new email notifications)
Twitter:   WebSocket (live tweet feed)
```

## Key Points
- HTTP = request-response, connection closes after each request
- **WebSocket** = persistent, two-way, real-time (best for chat, gaming)
- **SSE** = one-way server push (best for live feeds, notifications)
- **Long Polling** = connection held until data available (middle ground)
- WebSocket starts as HTTP then upgrades to persistent connection
