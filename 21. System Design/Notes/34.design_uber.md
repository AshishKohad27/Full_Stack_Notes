# 34. Design: Uber / Ride-Sharing

## Step 1: Requirements

```
Functional:
  - Rider requests a ride (pickup & drop location)
  - Match rider with nearby available driver
  - Real-time driver location tracking
  - Fare calculation
  - Payment processing
  - Trip history

Non-Functional:
  - Low latency (match driver < 5 seconds)
  - Real-time location updates (every 3-5 seconds)
  - High availability
  - Handle millions of concurrent rides
```

## Step 3: Core Flows

### Rider Requests a Ride
```
1. Rider → "I need a ride from A to B"
2. Ride Service creates ride request
3. Location Service → find all available drivers near point A
4. Matching Service → pick best driver (nearest + highest rating)
5. Send request to driver → Driver accepts/rejects
6. If accepted → ride starts, both see each other on map
7. Trip ends → fare calculated → payment processed
```

### Driver Location Updates (The Hard Part!)
```
Every driver sends location update every 3-5 seconds.

1 million active drivers × update every 4 sec = 250,000 updates/second!

How to handle?
  Driver App → WebSocket → Location Service → store in Redis/in-memory DB
  NOT in main database (too many writes!)

  Redis key: "driver:12345" → { lat: 19.07, lng: 72.87, available: true }
  TTL: 30 seconds (if no update, consider driver offline)
```

### Finding Nearby Drivers (Geospatial Query)
```
"Find all available drivers within 3 km of rider's location"

Option 1: GeoHash
  - Divide earth into grid cells
  - Each cell has a hash code
  - Nearby locations have similar hash codes
  - Store in Redis: GEOADD drivers 72.87 19.07 "driver123"
  - Query: GEORADIUS drivers 72.87 19.07 3 km

Option 2: QuadTree
  - Divide map into 4 quadrants recursively
  - Each leaf node contains drivers in that area
  - Efficient for spatial queries

Uber uses a combination of both!
```

## Step 4: Fare Calculation

```
Basic: fare = base_fare + (per_km × distance) + (per_min × time)

Surge Pricing:
  - More riders than drivers in area → price goes UP
  - surge_multiplier = demand / supply
  - fare = base_fare × surge_multiplier

  Normal:  demand=100, supply=100 → multiplier=1.0x → Rs.200
  Surge:   demand=500, supply=100 → multiplier=2.5x → Rs.500
```

## Step 5: Key Services

```
1. Ride Service      → manage ride lifecycle (request, match, start, end)
2. Location Service  → track real-time driver locations
3. Matching Service  → match rider with best driver
4. Pricing Service   → calculate fare + surge pricing
5. Payment Service   → process payments
6. Notification      → push notifications (ride accepted, driver arriving)
7. Trip History      → store completed trips
```

## Step 6: Tech Stack

```
Real-time location:  Redis GEO / in-memory store
Communication:       WebSocket (real-time tracking)
Database:            PostgreSQL (users, trips) + Cassandra (location history)
Queue:               Kafka (event processing)
Maps:                Google Maps API / Mapbox
Cache:               Redis (driver locations, ride state)
```

## Key Design Decisions
```
1. WebSocket for real-time location (not polling!)
2. Redis GEO for spatial queries (find nearby drivers)
3. Separate location data from main DB (too many writes)
4. Kafka for event-driven communication between services
5. Surge pricing based on real-time demand/supply ratio
```
