## The 10 Building Blocks (Memorize These!)

```
1. DNS              → Resolves domain to IP
2. Load Balancer    → Distributes traffic
3. API Gateway      → Single entry, auth, rate limit
4. Cache (Redis)    → Fast reads, reduce DB load
5. CDN              → Static content globally
6. Database         → Store data (SQL/NoSQL)
7. Message Queue    → Async processing
8. Object Storage   → Files, images, videos (S3)
9. Search Engine    → Full-text search (Elasticsearch)
10. Monitoring      → Track health, errors, performance
```

## Quick Decision Matrix

| Scenario                             | Solution                          |
|--------------------------------------|-----------------------------------|
| Need fast reads                      | Cache (Redis)                     |
| Database too large                   | Sharding                          |
| Too many reads on DB                 | Read replicas + Cache             |
| Users worldwide, high latency        | CDN + Multi-region               |
| Real-time updates                    | WebSocket                         |
| Background processing               | Message Queue (Kafka)             |
| Search across text                   | Elasticsearch                     |
| Store images/videos                  | S3 + CDN                         |
| Prevent overload                     | Rate Limiter                      |
| Service-to-service calls             | gRPC (internal), REST (external)  |
| Distributed transactions             | Saga Pattern                      |
| Prevent cascade failure              | Circuit Breaker                   |
| Need unique IDs at scale             | Snowflake ID / UUID               |
| User authentication                  | JWT + OAuth                       |
| Same client → same server            | Consistent Hashing                |

## Top 10 Interview Questions & One-Line Answers

```
Q1: How to handle millions of users?
A: Horizontal scaling + Load Balancer + Cache + CDN + Database sharding

Q2: How to make system highly available?
A: Redundancy at every layer + Multiple data centers + Auto-failover

Q3: SQL or NoSQL?
A: SQL for relationships/ACID, NoSQL for scale/flexibility. Most apps use BOTH.

Q4: How to handle real-time features?
A: WebSocket for two-way, SSE for one-way, Long Polling as fallback

Q5: How to prevent single point of failure?
A: Replicate everything: multiple servers, DB replicas, multi-region

Q6: Strong or Eventual consistency?
A: Strong for money/inventory, Eventual for social feeds/likes

Q7: How to handle traffic spikes?
A: Auto-scaling + Message Queue (absorb spike) + Cache + Rate Limiting

Q8: How to design for global users?
A: Multi-region deployment + CDN + Geographic DB sharding + DNS routing

Q9: How to handle large file uploads?
A: Pre-signed URL → direct upload to S3 → async processing via queue

Q10: Monolith or Microservices?
A: Start Monolith, migrate to Microservices when team/app grows large
```

## Template: Answer ANY System Design Question

```
"Design X" → Follow this template:

1. CLARIFY (2 min)
   "Who are the users? How many? What's most important feature?"

2. ESTIMATE (2 min)
   "DAU: 10M, Reads/sec: 1000, Storage: 500GB/year"

3. HIGH-LEVEL DESIGN (10 min)
   Draw: Client → LB → Services → DB/Cache
   List APIs: POST /create, GET /feed

4. DEEP DIVE (10 min)
   - Database choice (SQL vs NoSQL + why)
   - Caching strategy (what to cache, TTL)
   - Key algorithm (feed ranking, matching, etc.)

5. SCALE & RELIABILITY (5 min)
   - "What if 10x traffic?" → auto-scale, cache, queue
   - "What if server crashes?" → redundancy, failover
   - "What about data loss?" → replication, backups
   - Trade-offs: CAP theorem choice
```

## Interview Day Reminders

```
1. START with questions, not answers
2. DRAW diagrams (boxes + arrows)
3. TALK about trade-offs (no perfect solution)
4. MENTION specific tools (Redis, Kafka, S3 - not just "cache", "queue")
5. ADDRESS failures ("What if X goes down?")
6. DON'T over-engineer (start simple, add complexity when asked)
7. THINK OUT LOUD (interviewer wants to see your thought process)
8. KNOW the numbers (latency, storage, throughput estimates)
```
