# 26. Database Normalization vs Denormalization

## Normalization (Split Data, Remove Duplication)

- Organizing data to **reduce redundancy** (no repeated data)
- Data split into **multiple related tables**

### Example: Without Normalization (BAD)

```
Orders Table (duplicated data everywhere!):
+----+--------+-------+---------+--------------+-----------+
| id | user   | email | product | product_price| order_date|
+----+--------+-------+---------+--------------+-----------+
| 1  | Ashish | a@m   | iPhone  | 80000        | 2024-01-01|
| 2  | Ashish | a@m   | Case    | 500          | 2024-01-02|
| 3  | Rahul  | r@m   | iPhone  | 80000        | 2024-01-03|
+----+--------+-------+---------+--------------+-----------+

Problems:
  - "Ashish" and "a@m" repeated (waste storage)
  - "iPhone" and "80000" repeated
  - If Ashish changes email → update EVERY row! (update anomaly)
  - If we delete order 3 → we lose Rahul's info! (delete anomaly)
```

### After Normalization (GOOD)

```
Users Table:
+----+--------+-------+
| id | name   | email |
+----+--------+-------+
| 1  | Ashish | a@m   |
| 2  | Rahul  | r@m   |
+----+--------+-------+

Products Table:
+----+---------+-------+
| id | name    | price |
+----+---------+-------+
| 1  | iPhone  | 80000 |
| 2  | Case    | 500   |
+----+---------+-------+

Orders Table:
+----+---------+------------+------------+
| id | user_id | product_id | order_date |
+----+---------+------------+------------+
| 1  | 1       | 1          | 2024-01-01 |
| 2  | 1       | 2          | 2024-01-02 |
| 3  | 2       | 1          | 2024-01-03 |
+----+---------+------------+------------+

No duplication! Change email once in Users table → reflected everywhere!
```

## Denormalization (Combine Data for Speed)

- Intentionally **add redundancy** back for **faster reads**
- Avoid expensive JOINs by duplicating data

### Why Denormalize?

```
Normalized (3 JOINs needed):
  SELECT users.name, products.name, orders.date
  FROM orders
  JOIN users ON orders.user_id = users.id
  JOIN products ON orders.product_id = products.id;

  3 table JOIN = SLOW at scale (millions of rows)!

Denormalized (no JOINs needed):
  SELECT user_name, product_name, order_date
  FROM orders_denormalized;

  1 table, no JOIN = FAST!
```

### Denormalized Table
```
Orders_Denormalized:
+----+--------+-------+---------+-------+------------+
| id | user   | email | product | price | order_date |
+----+--------+-------+---------+-------+------------+
| 1  | Ashish | a@m   | iPhone  | 80000 | 2024-01-01 |
| 2  | Ashish | a@m   | Case    | 500   | 2024-01-02 |
+----+--------+-------+---------+-------+------------+

Yes, data is duplicated. But reads are MUCH faster!
```

## Comparison

| Feature           | Normalization              | Denormalization            |
|-------------------|----------------------------|----------------------------|
| Data duplication  | No duplication             | Intentional duplication    |
| Read speed        | Slower (needs JOINs)       | Faster (no JOINs)         |
| Write speed       | Faster (update one place)  | Slower (update many places)|
| Storage           | Less                       | More                       |
| Data consistency  | Easy to maintain           | Harder (sync duplicates)   |
| Best for          | Write-heavy, small scale   | Read-heavy, large scale    |

## When to Use What?

```
Normalize when:
  ✅ Data changes frequently (user updates email often)
  ✅ Write-heavy system
  ✅ Data integrity is critical (banking)
  ✅ Storage is expensive

Denormalize when:
  ✅ Read-heavy system (100 reads per 1 write)
  ✅ Need fast queries at scale
  ✅ Data doesn't change often
  ✅ Can tolerate slight inconsistency
```

## Real World

```
Instagram:
  - User profile: Normalized (data changes - email, bio)
  - Feed/Timeline: Denormalized (read-heavy, pre-computed)

Amazon:
  - Product catalog: Normalized (price changes, stock updates)
  - Order history: Denormalized (rarely changes, read frequently)
```

## Key Points
- **Normalize** = split tables, no duplication, JOINs needed
- **Denormalize** = combine tables, duplication OK, no JOINs
- Most real systems use **BOTH**: normalized for writes, denormalized views for reads
- This is basically what **CQRS** pattern does (separate read/write models)
