# 27. Idempotency

- Idempotent = doing something **multiple times gives SAME result** as doing it once
- Critical in distributed systems where requests can be retried

## Real World Analogy

```
IDEMPOTENT:
  Pressing elevator button 10 times = same as pressing once
  Elevator comes once, not 10 times.

NOT IDEMPOTENT:
  Ordering pizza by calling 10 times = you get 10 pizzas!
  Each call creates a NEW order.
```

## Why It Matters

```
Scenario: User clicks "Pay Now" button

1. Request sent to Payment Service
2. Payment processed successfully
3. Response going back to user... NETWORK TIMEOUT!
4. User didn't get confirmation → clicks "Pay Now" AGAIN
5. Without idempotency → User is CHARGED TWICE! Rs.2000 instead of Rs.1000!

With idempotency:
  First click → Payment processed (Rs.1000 charged)
  Second click → "Already processed!" → No duplicate charge
```

## How to Implement

```
Use an IDEMPOTENCY KEY (unique ID for each operation):

Request 1:
  POST /payment
  Idempotency-Key: "abc-123-xyz"
  { amount: 1000 }
  → Server processes → stores key "abc-123-xyz" = completed

Request 2 (retry/duplicate):
  POST /payment
  Idempotency-Key: "abc-123-xyz"    ← SAME key!
  { amount: 1000 }
  → Server checks: "abc-123-xyz" already processed!
  → Returns cached response → NO duplicate payment!
```

## HTTP Methods & Idempotency

| Method   | Idempotent? | Why                                         |
|----------|-------------|---------------------------------------------|
| GET      | Yes         | Reading same data 10 times = same result    |
| PUT      | Yes         | Setting value to 5 ten times = still 5      |
| DELETE   | Yes         | Deleting same item 10 times = item gone     |
| POST     | NO          | Creating order 10 times = 10 orders!        |
| PATCH    | It depends  | "Set age=25" is idempotent, "age+1" is not  |

```
Example:
  GET /user/123        → always returns same user (idempotent)
  PUT /user/123 {age:25} → age is 25 no matter how many times (idempotent)
  DELETE /user/123     → user deleted, calling again = already gone (idempotent)
  POST /orders         → creates NEW order each time (NOT idempotent!)
```

## Key Points
- Idempotent = safe to retry without side effects
- Critical for: payments, orders, any money-related operation
- Use **idempotency keys** to prevent duplicate processing
- GET, PUT, DELETE are naturally idempotent
- POST is NOT idempotent (must add idempotency key manually)
- Stripe, Razorpay, PayPal all support idempotency keys
