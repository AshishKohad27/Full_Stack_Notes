# 19. Database Indexing & Performance

- Index = a **shortcut** to find data faster (like an index in a book)
- Without index: database scans EVERY row (slow)
- With index: database jumps directly to the right row (fast)

## Without Index vs With Index

```
WITHOUT Index (Full Table Scan):
  "Find user with email = ashish@mail.com"
  → Check row 1... no
  → Check row 2... no
  → Check row 3... no
  → ... check ALL 10 million rows
  → Found at row 7,894,231!
  Time: SLOW (scans everything)

WITH Index on email column:
  "Find user with email = ashish@mail.com"
  → Index lookup → directly go to row 7,894,231
  Time: FAST (like finding a word in dictionary using index)
```

## How Index Works (Simplified)

```
Think of a BOOK:

Without Index:
  Find "Caching" → flip through every page → SLOW

With Index (back of book):
  A... B... Caching → Page 87 → go directly! → FAST

Database index works the same way.
It creates a sorted data structure (B-Tree) that points to actual rows.
```

## Types of Indexes

### 1. Primary Index (Primary Key)
```
Automatically created on Primary Key column.
Every table should have one.

CREATE TABLE users (
    id INT PRIMARY KEY,  -- index automatically created!
    name VARCHAR(100),
    email VARCHAR(100)
);
```

### 2. Secondary Index (On other columns)
```
You create these manually on columns you query often.

CREATE INDEX idx_email ON users(email);
-- Now: SELECT * FROM users WHERE email = 'ashish@mail.com' is FAST!

CREATE INDEX idx_name ON users(name);
-- Now: SELECT * FROM users WHERE name = 'Ashish' is FAST!
```

### 3. Composite Index (Multiple columns)
```
CREATE INDEX idx_name_age ON users(name, age);
-- Fast for: WHERE name = 'Ashish' AND age = 22
-- Fast for: WHERE name = 'Ashish' (uses first column)
-- NOT fast for: WHERE age = 22 (doesn't use first column!)

ORDER MATTERS! Index on (A, B) helps queries on (A) and (A,B) but NOT (B) alone.
```

## Index Trade-offs

```
PROS:
  + SELECT queries become MUCH faster
  + Finding specific rows is instant

CONS:
  - INSERT/UPDATE/DELETE become slower (index must be updated too)
  - Takes extra storage space
  - Too many indexes = slow writes

RULE: Index columns you SEARCH by often, not every column.
```

## Database Performance Tips

```
1. INDEX wisely
   - Index columns used in WHERE, JOIN, ORDER BY
   - Don't index columns with low cardinality (e.g., boolean gender)

2. QUERY optimization
   - SELECT only needed columns (not SELECT *)
   - Use LIMIT for pagination
   - Avoid SELECT * FROM large_table

3. CONNECTION pooling
   - Reuse database connections instead of creating new ones
   - Tools: HikariCP (Java), pgBouncer (PostgreSQL)

4. DENORMALIZATION
   - Sometimes duplicate data to avoid expensive JOINs
   - Trade storage for speed

5. READ REPLICAS
   - Route read queries to slave databases
   - Master handles writes only

6. CACHING
   - Cache frequent query results in Redis
   - Don't hit database for same query 1000 times
```

## N+1 Query Problem -- INTERVIEW FAVORITE!

```
BAD (N+1 queries):
  SELECT * FROM orders;              -- 1 query → 100 orders
  For each order:
    SELECT * FROM users WHERE id = order.user_id;  -- 100 queries!
  Total: 101 queries! SLOW!

GOOD (1 query with JOIN):
  SELECT * FROM orders
  JOIN users ON orders.user_id = users.id;  -- 1 query! FAST!
  Total: 1 query!
```

## Key Points
- Index = sorted lookup structure for faster reads
- Always index columns used in WHERE and JOIN
- Too many indexes = slow writes (balance is key)
- Use EXPLAIN to analyze query performance
- N+1 problem: use JOINs instead of loops
- Real apps use: indexes + caching + read replicas together
