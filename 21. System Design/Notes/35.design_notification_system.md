# 35. Design: Notification System

## Step 1: Requirements

```
Functional:
  - Send push notifications (mobile)
  - Send SMS
  - Send email
  - Support different types: marketing, transactional, system alerts
  - User notification preferences (opt-in/opt-out)
  - Notification history

Non-Functional:
  - High throughput (millions of notifications/day)
  - At-least-once delivery
  - Low latency for critical notifications (OTP < 5 sec)
  - No duplicate notifications
```

## Step 2: High-Level Design

```
Triggers:                  ┌────────────────────┐
  Order placed ──────────→ │                    │     ┌─────────────┐
  Payment done ──────────→ │  Notification      │────→│ Push (FCM)  │
  New follower ──────────→ │  Service           │     └─────────────┘
  OTP request  ──────────→ │                    │     ┌─────────────┐
  Marketing    ──────────→ │                    │────→│ SMS (Twilio)│
                           └─────────┬──────────┘     └─────────────┘
                                     │                ┌─────────────┐
                                     └───────────────→│ Email(SES)  │
                                                      └─────────────┘
```

## Step 3: Detailed Flow

```
1. Event occurs: "Ashish placed an order"

2. Event → Message Queue (Kafka)
   Topic: "order.placed"
   Data: { user_id: 123, order_id: 456, amount: 1500 }

3. Notification Service picks up event:
   a. Check user preferences: "Does Ashish want order notifications?" → YES
   b. Check channels: Push + Email enabled, SMS disabled
   c. Build notification content from template:
      Title: "Order Confirmed!"
      Body: "Your order #456 of Rs.1500 is confirmed."

4. Send to respective channels:
   → Push Queue → Push Worker → FCM (Firebase) → Ashish's phone
   → Email Queue → Email Worker → SES (Amazon) → ashish@mail.com

5. Track delivery status:
   → Push: delivered/failed
   → Email: sent/opened/bounced
   → SMS: delivered/failed

6. Store in notification history DB
```

## Step 4: Preventing Duplicates

```
Problem: Network issues → same notification sent twice

Solution: Idempotency!
  Each notification has unique ID: notif_id = "order_456_push_123"

  Before sending:
    Check Redis: "Have I sent notif_id already?"
    YES → skip (don't send again)
    NO → send and mark as sent in Redis
```

## Step 5: Rate Limiting & Throttling

```
Don't spam users!

Rules:
  - Max 3 marketing notifications per day per user
  - Max 1 SMS per minute per user (OTP exceptions)
  - Quiet hours: No marketing notifications 10 PM - 8 AM

  if (marketing_count_today >= 3) → skip
  if (current_hour < 8 || current_hour > 22) → delay to morning
```

## Step 6: Priority Queue

```
Not all notifications are equal!

HIGH priority:  OTP, payment alerts, security alerts
  → Separate fast queue → processed immediately

MEDIUM priority: Order updates, delivery status
  → Normal queue

LOW priority: Marketing, recommendations
  → Batch queue → processed during off-peak hours

High priority queue is NEVER blocked by marketing emails!
```

## Key Components
```
1. Message Queue (Kafka)     → decouple triggers from sending
2. Priority Queues           → critical first, marketing later
3. Template Service          → reusable notification templates
4. Preference Service        → user opt-in/opt-out
5. Rate Limiter              → prevent spamming
6. Delivery Tracker          → track sent/delivered/failed
7. Push: FCM (Android), APNs (iOS)
8. SMS: Twilio, AWS SNS
9. Email: AWS SES, SendGrid
```
