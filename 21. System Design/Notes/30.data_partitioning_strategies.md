# 30. Data Partitioning Strategies

- Partitioning = splitting data across multiple databases/servers
- Different from replication (copy) — partitioning means each part has DIFFERENT data

## Types of Partitioning

### 1. Horizontal Partitioning (Sharding)
```
SAME table split by ROWS across servers:

Server 1: Users with ID 1 - 1,000,000
Server 2: Users with ID 1,000,001 - 2,000,000
Server 3: Users with ID 2,000,001 - 3,000,000

Each server has same columns but different rows.
Most common approach!
```

### 2. Vertical Partitioning
```
SAME table split by COLUMNS across servers:

Server 1 (User Basic):  id, name, email
Server 2 (User Profile): id, bio, profile_pic, cover_photo
Server 3 (User Settings): id, theme, language, notifications

Frequently accessed data (name, email) on fast server.
Heavy data (photos) on storage-optimized server.
```

### 3. Directory-Based Partitioning
```
A lookup service tells you WHERE data lives:

Client → "Where is User 12345?"
Directory → "Server 3"
Client → Server 3 → get user data

Directory = mapping table:
  User 1-1000    → Server 1
  User 1001-2000 → Server 2
  User 2001-3000 → Server 3

Flexible! Can move data without changing client code.
```

## Partitioning Criteria

```
1. Range-Based:    Users A-M → Shard 1, N-Z → Shard 2
2. Hash-Based:     hash(user_id) % num_shards → determines shard
3. Geographic:     India users → India DB, US users → US DB
4. Time-Based:     2024 data → DB1, 2025 data → DB2
```

## Key Points
- **Horizontal** = split rows (most common, called sharding)
- **Vertical** = split columns (separate heavy data)
- **Hash-based** gives best even distribution
- **Geographic** partitioning = data closer to users = faster
- Always plan for **re-partitioning** (adding more shards as data grows)
