# 9. Monolith vs Microservices

## Monolith Architecture
- Entire application is **ONE single unit**
- All features (auth, payments, orders, notifications) in ONE codebase, ONE deployment
- Like a **single big building** where everything is under one roof

```
+------------------------------------------+
|            MONOLITH APPLICATION           |
|                                          |
|  ┌──────┐ ┌──────┐ ┌──────┐ ┌────────┐  |
|  | Auth | | Order| | Pay  | | Notify |  |
|  └──────┘ └──────┘ └──────┘ └────────┘  |
|                                          |
|           ONE Database                   |
|           ONE Deployment                 |
|           ONE Codebase                   |
+------------------------------------------+
```

## Microservices Architecture
- Application is **broken into small independent services**
- Each service does ONE thing, has its OWN database, deployed separately
- Like **separate shops** in a mall - each runs independently

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ Auth     │  │ Order    │  │ Payment  │  │ Notify   │
│ Service  │  │ Service  │  │ Service  │  │ Service  │
│          │  │          │  │          │  │          │
│ Own DB   │  │ Own DB   │  │ Own DB   │  │ Own DB   │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
     │              │              │              │
     └──────────────┴──────────────┴──────────────┘
                    communicate via APIs
```

## Comparison

| Feature           | Monolith                       | Microservices                  |
|-------------------|--------------------------------|--------------------------------|
| Codebase          | One big codebase               | Many small codebases           |
| Deployment        | Deploy everything together     | Deploy each service separately |
| Scaling           | Scale ENTIRE app               | Scale ONLY the busy service    |
| Technology        | One tech stack for all         | Each service can use different |
| Team              | One big team                   | Small teams per service        |
| Complexity        | Simple to start                | Complex to manage              |
| Failure           | One bug can crash EVERYTHING   | One service fails, others work |
| Communication     | Function calls (fast)          | Network calls/APIs (slower)    |
| Best for          | Small apps, startups, MVPs     | Large apps, big teams          |

## Real World Analogy

```
MONOLITH = Restaurant with 1 chef who cooks everything
  - Simple to run
  - If chef is sick, restaurant is closed
  - Hard to scale (can't add "half a chef")

MICROSERVICES = Food court with separate stalls
  - Pizza stall, Burger stall, Juice stall
  - If Pizza stall closes, Burger stall still works
  - Busy Saturday? Add more staff to ONLY the busy stall
  - Each stall can have different cooking style
```

## When to Use What?

```
Start with MONOLITH when:
  - Small team (< 10 developers)
  - New product / MVP (just starting)
  - Simple application
  - Need to move fast

Move to MICROSERVICES when:
  - Team is growing (50+ developers)
  - App is too big to manage as one unit
  - Different parts need to scale independently
  - Need to deploy features independently
```

## Real Companies

```
Started as Monolith → Moved to Microservices:
  - Netflix (streaming became too complex)
  - Amazon (teams stepping on each other's code)
  - Uber (needed independent scaling)

Still Monolith (and it works!):
  - Basecamp / Hey.com
  - Shopify (modular monolith)
```

## Key Points
- Monolith = simple, good to START with
- Microservices = complex, good to SCALE with
- Don't start with microservices unless you have a big team
- **"Monolith First"** - Martin Fowler (famous architect)
- Microservices communicate via **APIs, Message Queues, or gRPC**
