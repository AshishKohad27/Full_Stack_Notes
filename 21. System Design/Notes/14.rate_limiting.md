# 14. Rate Limiting

- Rate Limiting = **controlling how many requests** a user can make in a time period
- Protects your system from abuse, DDoS attacks, and overload

## Why Rate Limiting?

```
WITHOUT Rate Limiting:
  - Bot sends 1,000,000 requests/second → your server CRASHES
  - User keeps refreshing → wastes server resources
  - Hacker tries 10,000 passwords → brute force attack succeeds

WITH Rate Limiting:
  - Max 100 requests/minute per user
  - Request 101 → "429 Too Many Requests" → blocked!
  - Server stays healthy, legitimate users are served
```

## Common Rate Limiting Algorithms

### 1. Fixed Window
```
Window: 1 minute, Limit: 100 requests

12:00:00 - 12:01:00 → up to 100 requests allowed
12:01:00 - 12:02:00 → counter resets, 100 more allowed

Problem: User sends 100 requests at 12:00:59 and 100 at 12:01:01
         = 200 requests in 2 seconds! (burst at window boundary)
```

### 2. Sliding Window
```
Looks at the LAST 60 seconds from current time (not fixed boundaries)

At 12:01:30, it checks requests from 12:00:30 to 12:01:30
No boundary burst problem!

More accurate but slightly more complex.
```

### 3. Token Bucket (Most Popular)
```
Bucket holds tokens. Each request uses 1 token. Tokens refill at fixed rate.

Bucket capacity: 10 tokens
Refill rate: 1 token/second

Second 0:  10 tokens available
User sends 5 requests → 5 tokens left
Second 5:  5 + 5 = 10 tokens (refilled)
User sends 10 requests → 0 tokens
Next request → BLOCKED (no tokens!)
Wait 1 second → 1 token → can send 1 request

Allows short bursts (up to bucket size) but limits sustained rate.
```

### 4. Leaky Bucket
```
Requests enter a bucket. Bucket "leaks" (processes) at fixed rate.

Like a funnel:
  - Pour water fast → funnel processes at constant rate
  - If you pour too much → water overflows (request rejected)

Smooths out traffic to a constant rate.
```

## Where to Implement?

```
1. API Gateway level (most common)
   → Rate limit before request reaches your services

2. Application level
   → In your code using libraries

3. Load Balancer level
   → Nginx can do basic rate limiting

4. Cloud level
   → AWS API Gateway, Cloudflare has built-in rate limiting
```

## Real World Examples

```
Twitter API:   300 requests per 15 minutes per user
GitHub API:    5000 requests per hour per user
Google Maps:   25,000 requests per day (free tier)
Instagram:     200 API calls per hour
```

## HTTP Response When Rate Limited

```
HTTP/1.1 429 Too Many Requests
Retry-After: 60

{
    "error": "Rate limit exceeded",
    "message": "Try again in 60 seconds"
}
```

## Key Points
- Rate limiting protects servers from overload and abuse
- **Token Bucket** is most commonly used algorithm
- Implement at **API Gateway** level for microservices
- Return **429 status code** with `Retry-After` header
- Rate limit by: IP address, user ID, API key
