# 31. Design: URL Shortener (bit.ly)

## Step 1: Requirements

```
Functional:
  - Shorten a long URL → short URL (e.g., bit.ly/abc123)
  - Redirect short URL → original URL
  - Optional: custom short URL, expiry, analytics

Non-Functional:
  - Low latency (redirect in < 100ms)
  - Highly available (99.99%)
  - Short URL should not be predictable (security)

Scale:
  - 100 million URLs created per month
  - 10:1 read to write ratio (1 billion redirects/month)
  - Store for 5 years
```

## Step 2: Estimation

```
URLs/month:  100M writes, 1B reads
URLs/second: writes = 100M / (30×24×3600) ≈ 40/sec
             reads = 1B / (30×24×3600) ≈ 400/sec

Storage: Each URL = ~500 bytes
  100M × 12 months × 5 years × 500 bytes = 300 TB
```

## Step 3: High-Level Design

```
                    ┌─────────────┐
  Client ──────────→│ API Gateway │
                    └──────┬──────┘
                           │
                    ┌──────┴──────┐
                    │ URL Service │
                    └──────┬──────┘
                     ┌─────┴─────┐
                     │           │
                  ┌──┴──┐   ┌───┴──┐
                  │Cache│   │  DB  │
                  │Redis│   │NoSQL │
                  └─────┘   └──────┘

APIs:
  POST /shorten  { long_url: "https://..." }  → { short_url: "abc123" }
  GET  /abc123   → 301 Redirect to long_url
```

## Step 4: Short URL Generation

```
Option A: Hash + Take first 7 chars
  MD5("https://long-url.com") → "a1b2c3d4e5f6..." → take "a1b2c3d"
  Problem: Hash collision (two URLs get same short code)
  Solution: Check DB, if exists, append counter and re-hash

Option B: Base62 Encoding (PREFERRED)
  Auto-increment ID → Convert to Base62

  ID: 12345 → Base62 → "3d7"
  ID: 9999999 → Base62 → "FXsk"

  Base62 = a-z (26) + A-Z (26) + 0-9 (10) = 62 characters
  7 chars = 62^7 = 3.5 TRILLION unique URLs!

Option C: Pre-generated Keys
  Generate millions of unique keys in advance
  Store in a Key DB
  When URL comes → pick next available key → fast!
```

## Step 5: Database Schema

```
Table: urls
+----------+-----------+---------------------------+------------+
| short_url| long_url  | created_at                | expires_at |
+----------+-----------+---------------------------+------------+
| abc123   | https://..| 2024-01-01 10:00:00       | NULL       |
+----------+-----------+---------------------------+------------+

Database choice: NoSQL (DynamoDB/Cassandra)
  - Simple key-value lookup
  - Needs horizontal scaling
  - No complex queries needed
```

## Step 6: Redirect Flow

```
User clicks: bit.ly/abc123

1. Browser → GET /abc123 → API Gateway → URL Service
2. URL Service → Check Redis Cache
3. Cache HIT → return long URL → 301 Redirect
4. Cache MISS → Query DB → Store in cache → 301 Redirect

301 (Permanent Redirect) vs 302 (Temporary):
  301 → Browser caches it, next time browser redirects itself (less server load)
  302 → Browser always asks server (good for analytics/tracking clicks)
```

## Step 7: Scale

```
- Cache: Redis (top 20% URLs handle 80% traffic)
- DB: NoSQL with sharding by short_url hash
- Rate limiting: prevent abuse (max 100 URLs/hour per user)
- Multiple data centers for global availability
```
