# 7. Caching

- Cache = **temporary fast storage** for frequently accessed data
- Instead of querying database every time, store result in cache
- Like keeping your most-used books on your desk instead of going to the library

## Why Cache?

```
WITHOUT Cache:                        WITH Cache:
User → Server → Database (SLOW)       User → Server → Cache (FAST! in memory)
                ~100ms                                  ~1ms

If data not in cache (cache miss):
User → Server → Cache (miss) → Database → Store in Cache → Return
Next time same data is requested → Cache (hit) → Return instantly
```

## How Cache Works

```
1. User requests data
2. Server checks CACHE first
3. If found (CACHE HIT) → return immediately (fast!)
4. If not found (CACHE MISS) → query database → store in cache → return
5. Next request for same data → CACHE HIT
```

## Cache Strategies

### 1. Cache-Aside (Lazy Loading) -- Most Common
```
Read:
  1. Check cache
  2. If HIT → return from cache
  3. If MISS → read from DB → write to cache → return

Write:
  1. Write to database
  2. Delete/update cache

Best for: Read-heavy workloads
```

### 2. Write-Through
```
Write:
  1. Write to cache AND database at same time

Read:
  1. Always read from cache (data is always up-to-date)

Pro: Cache is always consistent with DB
Con: Every write is slower (writes to both places)
```

### 3. Write-Behind (Write-Back)
```
Write:
  1. Write to cache ONLY (instant!)
  2. Cache writes to database LATER (async)

Pro: Super fast writes
Con: Risk of data loss if cache crashes before writing to DB
```

## Cache Eviction Policies (When Cache is Full, What to Remove?)

| Policy | Full Form               | How It Works                           |
|--------|-------------------------|----------------------------------------|
| LRU    | Least Recently Used     | Remove item not accessed for longest   |
| LFU    | Least Frequently Used   | Remove item accessed least times       |
| FIFO   | First In First Out      | Remove oldest item                     |
| TTL    | Time To Live            | Remove after set time (e.g., 5 min)    |

```
Example - LRU with cache size 3:

Access: A B C D A B E

Cache state:
[A]         → A
[A, B]      → B
[A, B, C]   → C (cache full)
[B, C, D]   → D (A removed - least recently used)
[C, D, A]   → A (B removed)
[D, A, B]   → B (C removed)
[A, B, E]   → E (D removed)
```

## Where to Cache?

```
1. Browser Cache     → CSS, JS, images cached in browser
2. CDN Cache         → Static files cached at edge servers worldwide
3. Server Cache      → Application-level cache (Redis, Memcached)
4. Database Cache    → Query result cache built into database
```

## Popular Cache Tools

| Tool       | Type           | Used For                      |
|------------|----------------|-------------------------------|
| Redis      | In-memory DB   | Sessions, leaderboards, cache |
| Memcached  | In-memory cache| Simple key-value caching      |

### Redis Example
```
SET user:123 '{"name":"Ashish","age":22}'   → store in cache
GET user:123                                 → retrieve from cache
EXPIRE user:123 3600                         → auto-delete after 1 hour
DEL user:123                                 → manually delete
```

## Cache Problems

```
1. Cache Stampede / Thundering Herd
   - Cache expires, 1000 users hit database simultaneously
   - Solution: Lock - only 1 request goes to DB, others wait for cache

2. Cache Inconsistency
   - Database updated but cache has old data
   - Solution: TTL (auto-expire) or update cache on write

3. Cache Penetration
   - Requests for data that doesn't exist (always cache miss)
   - Solution: Cache null results too (with short TTL)
```

## Key Points
- Cache makes reads **100x faster** (memory vs disk)
- Use cache for **read-heavy** data that doesn't change often
- **Redis** is the most popular caching solution
- LRU is the most common eviction policy
- Always set **TTL** to avoid stale (outdated) data
- Cache is NOT permanent storage - it can be lost
