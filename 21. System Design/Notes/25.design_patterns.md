# 25. System Design Patterns

- Patterns = **proven solutions** to common problems

## 1. Circuit Breaker Pattern

### Problem
```
Service A calls Service B. Service B is DOWN.
Service A keeps retrying → wastes resources → A also slows down → CASCADING FAILURE!

Payment Service → Notification Service (DOWN)
Payment Service keeps trying → Payment Service also becomes slow → EVERYTHING CRASHES
```

### Solution: Circuit Breaker
```
Like an electrical circuit breaker - cuts the connection when something is wrong.

3 States:
  CLOSED (normal):
    → Requests flow normally
    → If failures > threshold → switch to OPEN

  OPEN (broken):
    → ALL requests immediately fail (don't even try)
    → Return fallback response
    → After timeout → switch to HALF-OPEN

  HALF-OPEN (testing):
    → Allow FEW requests through
    → If they succeed → back to CLOSED
    → If they fail → back to OPEN

Example:
  Payment Service → Notification Service (failing)

  Closed: tries 5 times, all fail → threshold reached!
  Open: stops calling Notification Service, returns "notification queued for later"
  After 30 sec → Half-Open: tries 1 request
  Success → Closed (back to normal!)
  Fail → Open (wait more)
```

## 2. CQRS (Command Query Responsibility Segregation)

### Problem
```
Read and Write operations have DIFFERENT needs:
  Writes: Need consistency, validation, complex logic
  Reads: Need speed, different data shapes, joins

Using ONE model for both = compromise on both
```

### Solution: Separate Read and Write
```
COMMAND (Write) side:              QUERY (Read) side:
  POST /orders ──→ Write DB        GET /orders ──→ Read DB
  (Normalized, ACID)               (Denormalized, fast)

  Write DB (MySQL) ──sync──→ Read DB (Redis/Elasticsearch)

Example - E-commerce:
  Write: Create order → Validate stock → Deduct payment → Save to MySQL
  Read: Show order history → Query from Redis (pre-computed, FAST)
```

```
Real Example - Twitter:
  Write: User posts tweet → Save in Write DB
  Read:  Show feed → Read from pre-computed feed cache

  Different databases optimized for different operations!
```

## 3. Saga Pattern (Distributed Transactions)

### Problem
```
In microservices, one operation may span MULTIPLE services:

"Place Order" needs:
  1. Order Service → Create order
  2. Payment Service → Charge money
  3. Inventory Service → Reduce stock
  4. Notification Service → Send email

What if Payment succeeds but Inventory fails?
  You charged the user but can't deliver! BAD!

Normal DB transaction (ACID) doesn't work across services!
```

### Solution: Saga Pattern
```
Each step has a COMPENSATING action (undo/rollback):

Step 1: Create Order        → Compensate: Cancel Order
Step 2: Reserve Payment     → Compensate: Refund Payment
Step 3: Reduce Inventory    → Compensate: Restore Inventory
Step 4: Send Notification   → Compensate: Send Cancellation

If Step 3 fails:
  → Compensate Step 2 (refund payment)
  → Compensate Step 1 (cancel order)
  → System is back to consistent state!
```

### Two Types of Saga

```
CHOREOGRAPHY (Event-based):
  Each service listens for events and acts:
  Order Created → Payment listens → charges → emits "Payment Done"
  Payment Done → Inventory listens → reduces stock → emits "Stock Reserved"

  No central coordinator. Services react to events.

ORCHESTRATION (Central coordinator):
  Saga Orchestrator tells each service what to do:
  Orchestrator → "Create order" → Order Service
  Orchestrator → "Charge payment" → Payment Service
  Orchestrator → "Reduce stock" → Inventory Service

  If any fails, Orchestrator runs compensations.
  Easier to manage, single point of control.
```

## 4. Event-Driven Architecture (EDA)

```
Services communicate by PUBLISHING and SUBSCRIBING to events.
Not "call this service" but "announce what happened".

Example - E-commerce:
  Order Service publishes: "OrderPlaced" event
    ↓
  Event Bus (Kafka)
    ↓
  Listeners:
    → Payment Service: "I'll process payment"
    → Inventory Service: "I'll update stock"
    → Email Service: "I'll send confirmation"
    → Analytics Service: "I'll track this"

Benefits:
  - Services are DECOUPLED (don't know about each other)
  - Easy to add new listeners (add Analytics without changing Order Service)
  - Async processing (user doesn't wait)
```

## 5. Event Sourcing

```
Instead of storing CURRENT state, store ALL EVENTS that happened.

Traditional:
  Account Balance = Rs.7000 (just the final number)

Event Sourcing:
  Event 1: Account Created (Rs.0)
  Event 2: Deposited Rs.10000
  Event 3: Withdrawn Rs.3000
  Event 4: Deposited Rs.5000
  Event 5: Withdrawn Rs.5000
  Current Balance = 0 + 10000 - 3000 + 5000 - 5000 = Rs.7000

Benefits:
  - Complete AUDIT TRAIL (know exactly what happened)
  - Can REPLAY events to rebuild state
  - Can go back in TIME (what was balance on March 1st?)
  - Used in: Banking, stock trading, accounting
```

## 6. Retry with Exponential Backoff

```
When a request fails, retry - but wait LONGER each time:

Attempt 1: Wait 1 second  → retry
Attempt 2: Wait 2 seconds → retry
Attempt 3: Wait 4 seconds → retry
Attempt 4: Wait 8 seconds → retry
Attempt 5: Give up → return error

WHY exponential?
  - If server is overloaded, immediate retries make it WORSE
  - Waiting longer gives server time to recover
  - Add "jitter" (random delay) so all clients don't retry at same time
```

## 7. Bulkhead Pattern

```
Isolate different parts so failure in ONE doesn't affect OTHERS.

Like compartments in a ship:
  If one compartment floods, others stay dry. Ship doesn't sink!

Example:
  Thread Pool for Payment API:    50 threads
  Thread Pool for Search API:     30 threads
  Thread Pool for Notification:   20 threads

  If Notification Service is slow and uses all its threads:
  → Payment and Search still work fine! (separate pools)

  Without Bulkhead:
  → All 100 threads could be stuck on Notification → EVERYTHING stops
```
