# 22. Object Storage & File Handling

- Object Storage = storing **files** (images, videos, PDFs) separately from your database
- Database is for structured data (name, email), NOT for big files
- Like a **warehouse** where you store boxes, and your office desk has the address of each box

## Why NOT Store Files in Database?

```
BAD: Store image directly in database
  - Database becomes HUGE → slow queries
  - Database backups become massive
  - Hard to serve files to millions of users

GOOD: Store file in Object Storage, save URL in database
  - Database stays small and fast
  - Files served directly from storage/CDN
  - Easy to scale independently
```

## How It Works

```
User uploads profile picture:

Step 1: Client → POST /upload (image file)
Step 2: Server → Saves file to Object Storage (S3)
Step 3: S3 → Returns file URL: "https://s3.aws.com/bucket/img123.jpg"
Step 4: Server → Saves URL in database: users.profile_pic = "https://s3.../img123.jpg"
Step 5: Server → Returns success to client

User views profile:
Step 1: Client → GET /profile
Step 2: Server → Returns { name: "Ashish", pic: "https://s3.../img123.jpg" }
Step 3: Browser → Loads image directly from S3/CDN (NOT from your server!)
```

```
Upload Flow:
  Client → Server → S3 (store file)
                   → DB (store URL)

Download Flow:
  Client → CDN/S3 → file loads directly (server not involved!)
```

## Pre-signed URLs (Secure Upload/Download)

```
Problem: You don't want anyone to upload files to YOUR S3 bucket!

Solution: Pre-signed URL (temporary permission)

Step 1: Client asks server: "I want to upload a photo"
Step 2: Server generates a pre-signed URL (valid for 5 minutes)
Step 3: Client uploads DIRECTLY to S3 using that URL
Step 4: URL expires after 5 minutes → no one else can use it

Benefits:
  - File goes directly to S3 (server doesn't handle large file)
  - Secure (URL expires quickly)
  - Server stays lightweight
```

## File Handling Architecture (Large Scale)

```
                                    ┌── S3 (store original)
User uploads video → Server → ─────┤
                                    └── Message Queue
                                          ↓
                                    Video Processing Service
                                    (compress, resize, thumbnail)
                                          ↓
                                    S3 (store processed versions)
                                          ↓
                                    CDN (serve globally)
                                          ↓
                                    User watches video FAST!
```

## Example: Instagram Photo Upload

```
1. User selects photo in app
2. App → Server: "I want to upload a photo"
3. Server → generates pre-signed S3 URL → sends to app
4. App → uploads photo directly to S3
5. S3 stores original photo
6. Message sent to queue: "New photo uploaded"
7. Image Processing Service picks up:
   - Creates thumbnail (150x150)
   - Creates medium (600x600)
   - Creates full size (1080x1080)
   - Applies filters
8. Processed images stored in S3
9. CDN caches them globally
10. Followers see photo loaded from nearest CDN server
```

## Key Points
- **Never store files in database** → use Object Storage (S3)
- Store file **URL** in database, file itself in S3
- Use **CDN** on top of S3 for global performance
- Use **pre-signed URLs** for secure direct uploads
- Process large files (video encoding) **asynchronously** via message queue
- S3 gives **99.999999999%** durability (11 nines!) → your file won't be lost
