# 38. Design: Rate Limiter (Detailed)

## Step 1: Requirements

```
Functional:
  - Limit requests per user/IP per time window
  - Support different limits for different APIs
  - Return 429 when limit exceeded
  - Distributed (works across multiple servers)

Non-Functional:
  - Low latency (rate check < 1ms)
  - Highly available
  - Accurate counting
```

## Step 2: Where to Place Rate Limiter?

```
Option 1: Client-side → Easy to bypass (user controls client) → BAD
Option 2: Server-side → In each service → duplicated logic
Option 3: API Gateway → Single place → BEST!

  Client → API Gateway (rate limit here!) → Services
```

## Step 3: Algorithm - Token Bucket (Deep Dive)

```
Config: bucket_size = 10, refill_rate = 2 tokens/second

Visualization:
  Time 0: Bucket [||||||||||] 10 tokens
  User sends 6 requests → uses 6 tokens
  Time 0: Bucket [||||      ] 4 tokens
  Time 1: Bucket [||||||    ] 6 tokens (+2 refilled)
  User sends 8 requests → only 6 allowed, 2 rejected!
  Time 1: Bucket [          ] 0 tokens
  Time 2: Bucket [||        ] 2 tokens (+2 refilled)

Why Token Bucket?
  - Allows short BURSTS (up to bucket size)
  - But limits SUSTAINED rate (refill rate)
  - Used by: AWS API Gateway, Stripe
```

## Step 4: Distributed Rate Limiting

```
Problem: 3 servers behind load balancer. User limit = 100/min.

Server 1 counts: 40 requests
Server 2 counts: 40 requests
Server 3 counts: 40 requests
Total: 120 requests! LIMIT EXCEEDED but no server knows!

Solution: Centralized counter in Redis

  Server 1 → Redis: INCR user:123:api_count → 41
  Server 2 → Redis: INCR user:123:api_count → 42
  Server 3 → Redis: INCR user:123:api_count → 43

  Redis key: "ratelimit:user123:minute:202401011030"
  TTL: 60 seconds (auto-expires)

  All servers check SAME Redis → accurate counting!
```

## Step 5: Redis Implementation

```
For each request:

1. key = "ratelimit:{user_id}:{api}:{current_minute}"
   Example: "ratelimit:user123:orders:202401011030"

2. count = Redis INCR key  (atomic increment)

3. If count == 1 → EXPIRE key 60  (set TTL for first request)

4. If count > limit → REJECT (return 429)
   Else → ALLOW (forward to service)

Redis commands:
  INCR ratelimit:user123:orders:202401011030    → returns current count
  EXPIRE ratelimit:user123:orders:202401011030 60  → auto-delete after 60s
```

## Step 6: Response Headers

```
HTTP/1.1 200 OK
X-RateLimit-Limit: 100         ← max requests allowed
X-RateLimit-Remaining: 67      ← requests remaining
X-RateLimit-Reset: 1704067200  ← when limit resets (Unix timestamp)

HTTP/1.1 429 Too Many Requests
Retry-After: 30                ← try again after 30 seconds
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
```

## Step 7: Different Rules

```
Rate limit rules stored in config/DB:

Rule 1: /api/login     → 5 requests / minute / IP   (prevent brute force)
Rule 2: /api/orders    → 100 requests / minute / user
Rule 3: /api/search    → 30 requests / minute / user
Rule 4: /api/upload    → 10 requests / hour / user   (heavy operation)

Free tier:  100 requests/day
Paid tier:  10,000 requests/day
Enterprise: unlimited
```

## Key Points
```
1. Place at API Gateway (single point of enforcement)
2. Use Redis for distributed counting (shared across servers)
3. Token Bucket algorithm (allows bursts + limits rate)
4. Return proper headers (Remaining, Reset, Retry-After)
5. Different limits for different APIs and user tiers
6. Always return 429 status code when rate limited
```
