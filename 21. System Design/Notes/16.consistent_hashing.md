# 16. Consistent Hashing

- Used to **distribute data across servers** efficiently
- When you add/remove a server, **minimal data is moved**
- Critical for: caching, database sharding, load balancing

## The Problem

```
Simple Hashing (Modulo):
  server = hash(key) % number_of_servers

  3 servers: hash("user1") % 3 = 1 → Server 1
             hash("user2") % 3 = 0 → Server 0
             hash("user3") % 3 = 2 → Server 2

PROBLEM: What if you ADD a 4th server?
  Now: hash("user1") % 4 = 2 → Server 2 (CHANGED!)
       hash("user2") % 4 = 3 → Server 3 (CHANGED!)
       hash("user3") % 4 = 1 → Server 1 (CHANGED!)

  ALMOST ALL keys are remapped! = cache miss for everyone = DB overload!
```

## Consistent Hashing Solution

```
Imagine a CIRCLE (hash ring) with values 0 to 360:

                    0/360
                   /     \
              Server A    Server B
             /                     \
           270                     90
             \                     /
              Server D    Server C
                   \     /
                    180

1. Hash each SERVER to a position on the ring
2. Hash each KEY to a position on the ring
3. Key goes to the FIRST server found clockwise

Key "user1" hashes to position 50 → goes clockwise → hits Server B (at 90)
Key "user2" hashes to position 200 → goes clockwise → hits Server D (at 270)
```

## When a Server is Added

```
Add Server E at position 150:

Before: Keys at 100-180 → Server C (at 180)
After:  Keys at 100-150 → Server E (NEW!)
        Keys at 150-180 → Server C (unchanged)

Only keys between Server D and Server E are remapped!
Other keys stay on the same server! MINIMAL disruption!
```

## When a Server is Removed

```
Remove Server B:

Before: Keys at 0-90 → Server B
After:  Keys at 0-90 → Server C (next clockwise)

Only Server B's keys are remapped to Server C.
All other keys stay where they are!
```

## Virtual Nodes (Even Distribution)

```
Problem: With few servers, distribution can be uneven.
         One server might get 60% of keys!

Solution: Each physical server gets MULTIPLE positions on the ring (virtual nodes)

Server A → positions 30, 120, 250 (3 virtual nodes)
Server B → positions 60, 180, 310 (3 virtual nodes)

More virtual nodes = more even distribution of keys.
```

## Where is Consistent Hashing Used?

```
1. Distributed Caches (Memcached, Redis Cluster)
   - Data distributed across cache servers
   - Adding/removing cache server moves minimal data

2. CDNs (Akamai, Cloudflare)
   - Route requests to nearest/correct edge server

3. Database Sharding
   - DynamoDB, Cassandra use consistent hashing for data distribution

4. Load Balancing
   - Same client always hits same server (sticky sessions)
```

## Key Points
- Regular hashing: add/remove server → almost ALL data moves (bad!)
- Consistent hashing: add/remove server → only **K/N** data moves (K=keys, N=servers)
- Virtual nodes ensure **even distribution** of data
- Used by: Amazon DynamoDB, Apache Cassandra, Memcached, Akamai CDN
