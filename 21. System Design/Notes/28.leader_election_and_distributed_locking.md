# 28. Leader Election & Distributed Locking

## A. Leader Election

### Problem
```
You have 5 servers. Some tasks should run on ONLY ONE server:
  - Sending daily email digest (send once, not 5 times!)
  - Running scheduled cleanup job
  - Coordinating data migration

Which server does it? They need to ELECT a leader!
```

### How It Works
```
5 Servers start up:
  Server 1: "I want to be leader!" → tries to acquire lock
  Server 2: "I want to be leader!" → tries to acquire lock
  Server 3: "I want to be leader!" → tries to acquire lock

  Server 1 wins! (got the lock first)
  Server 1 = LEADER (does the special task)
  Server 2,3,4,5 = FOLLOWERS (do normal work)

If Leader (Server 1) crashes:
  Servers 2-5 detect: "Leader is gone!" (heartbeat stopped)
  New election → Server 3 becomes new leader!
  System continues working!
```

### Example: Database Replication
```
3 Database servers:
  DB-1 (Leader/Master) → handles all WRITES
  DB-2 (Follower)      → handles READS
  DB-3 (Follower)      → handles READS

  DB-1 crashes!
  Leader Election happens:
  DB-2 elected as new Leader → now handles WRITES
  DB-3 stays Follower

  User didn't notice any downtime!
```

### Tools for Leader Election
```
- ZooKeeper (most popular for distributed coordination)
- etcd (used by Kubernetes)
- Consul (HashiCorp)
- Redis (using SETNX for simple cases)
```

## B. Distributed Locking

### Problem
```
Two servers try to update SAME data at SAME time:

Server 1: Read balance = Rs.1000
Server 2: Read balance = Rs.1000

Server 1: Deduct Rs.500 → Write Rs.500
Server 2: Deduct Rs.300 → Write Rs.700

Final balance: Rs.700 (WRONG! Should be Rs.200)

Server 2 overwrote Server 1's update! This is a RACE CONDITION.
```

### Solution: Distributed Lock
```
Before updating, ACQUIRE a lock. Only ONE server can hold it.

Server 1: Acquire lock ✅ (got it!)
Server 2: Acquire lock ❌ (locked! wait...)

Server 1: Read Rs.1000 → Deduct Rs.500 → Write Rs.500
Server 1: Release lock

Server 2: Acquire lock ✅ (now available!)
Server 2: Read Rs.500 → Deduct Rs.300 → Write Rs.200
Server 2: Release lock

Final balance: Rs.200 ✅ CORRECT!
```

### Redis Distributed Lock Example
```
Lock:
  SET lock:payment "server1" NX EX 30
  NX = only set if NOT exists (atomic)
  EX 30 = auto-expire in 30 seconds (safety: if server crashes, lock auto-releases)

Unlock:
  DEL lock:payment

If Server 1 crashes while holding lock:
  → Lock auto-expires after 30 seconds
  → Other servers can acquire it
  → No deadlock!
```

### Redlock Algorithm (Safer)
```
Problem: Single Redis = Single Point of Failure

Redlock uses 5 independent Redis instances:
  1. Try to acquire lock on ALL 5 Redis instances
  2. If lock acquired on 3+ (majority) → lock is acquired!
  3. If lock acquired on < 3 → release all, retry

Even if 2 Redis instances crash, lock still works!
```

## Heartbeat & Health Check

```
How do servers know if another server is alive?

HEARTBEAT:
  Every server sends "I'm alive!" signal every 5 seconds.

  Server 1 → "ping" → Coordinator
  Server 2 → "ping" → Coordinator
  Server 3 → "....."  (no ping for 15 seconds)

  Coordinator: "Server 3 is DEAD! Remove from load balancer!"

HEALTH CHECK:
  Load balancer ASKS each server: "Are you OK?"

  LB → /health → Server 1 → { status: "healthy" } ✅
  LB → /health → Server 2 → { status: "healthy" } ✅
  LB → /health → Server 3 → TIMEOUT ❌ → stop sending traffic
```

## Key Points
- **Leader Election** = one server is chosen for special tasks
- **Distributed Lock** = prevent race conditions across servers
- **Heartbeat** = servers tell coordinator they're alive
- Use **Redis SETNX** for simple distributed locks
- Use **ZooKeeper/etcd** for leader election
- Always set **lock expiry** to prevent deadlocks
