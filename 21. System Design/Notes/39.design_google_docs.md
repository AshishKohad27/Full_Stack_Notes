# 39. Design: Google Docs (Real-Time Collaboration)

## Step 1: Requirements

```
Functional:
  - Create/edit documents
  - Real-time collaboration (multiple users edit simultaneously)
  - See other users' cursors and changes live
  - Version history (undo, view past versions)
  - Share with permissions (view/edit/comment)
  - Auto-save

Non-Functional:
  - Real-time sync (< 100ms latency)
  - No data loss
  - Handle conflicts (two users edit same line)
  - Works offline (sync when back online)
```

## Step 2: The Hard Problem - Conflict Resolution

```
Ashish and Rahul both editing same document:

Document: "Hello World"

Ashish types "Java " at position 6:  "Hello Java World"
Rahul  deletes "World" at position 6: "Hello "

Both happen at SAME time!

Without conflict resolution:
  Result could be: "Hello Java " or "Hello World" → INCONSISTENT!

With conflict resolution (OT or CRDT):
  Both changes applied correctly: "Hello Java " → CONSISTENT everywhere!
```

## Step 3: Conflict Resolution Approaches

### Operational Transformation (OT) - Used by Google Docs
```
Transforms operations based on what happened before:

1. Ashish: Insert "Java " at position 6
2. Rahul: Delete char at position 6

Server receives both:
  → Ashish's insert came first → applied: "Hello Java World"
  → Rahul's delete: originally position 6, but Ashish inserted 5 chars
  → Transform: position 6 → now position 11 (shifted by 5)
  → Delete at position 11: "Hello Java Worl"

Both Ashish and Rahul see the same final result!
Server TRANSFORMS operations to account for concurrent changes.
```

### CRDT (Conflict-free Replicated Data Types) - Used by Figma
```
Data structure that automatically resolves conflicts.
Every character has a unique ID and position between neighbors.

Like assigning decimal positions:
  H(0.1) e(0.2) l(0.3) l(0.4) o(0.5)

Ashish inserts "X" between l and o:
  H(0.1) e(0.2) l(0.3) l(0.4) X(0.45) o(0.5)

Rahul inserts "Y" between l and o:
  H(0.1) e(0.2) l(0.3) l(0.4) Y(0.42) o(0.5)

No conflict! Both get merged:
  H(0.1) e(0.2) l(0.3) l(0.4) Y(0.42) X(0.45) o(0.5)
```

## Step 4: High-Level Architecture

```
┌──────────┐                ┌───────────────┐
│ Ashish   │──WebSocket────→│               │
│ Browser  │←──────────────│  Collaboration │──→ Document DB
└──────────┘               │  Server        │
                           │               │──→ Redis (active sessions)
┌──────────┐               │               │
│ Rahul    │──WebSocket────→│               │──→ Version History DB
│ Browser  │←──────────────│               │
└──────────┘               └───────────────┘
```

## Step 5: Real-Time Sync Flow

```
1. Ashish opens doc → WebSocket connection to Collaboration Server
2. Rahul opens same doc → WebSocket connection to same server
3. Server knows: "Doc ABC has 2 users: Ashish, Rahul"

4. Ashish types "Hello":
   → Operation sent via WebSocket: { type: "insert", text: "Hello", pos: 0 }
   → Server applies OT → stores in DB
   → Server broadcasts to Rahul via WebSocket
   → Rahul's editor shows "Hello" appearing

5. Auto-save every few seconds → persist to database
```

## Step 6: Version History

```
Every change is stored as an OPERATION LOG:

Version 1: Document created (empty)
Version 2: Insert "Hello" at pos 0 (by Ashish, 10:00 AM)
Version 3: Insert " World" at pos 5 (by Rahul, 10:01 AM)
Version 4: Delete "World" (by Ashish, 10:02 AM)
Version 5: Insert "Java" at pos 6 (by Rahul, 10:03 AM)

Current document: "Hello Java"

User can go back to any version (like git for documents!)
Implemented using Event Sourcing pattern.
```

## Key Design Decisions
```
1. WebSocket for real-time collaboration
2. OT (Operational Transformation) for conflict resolution
3. Event Sourcing for version history
4. Redis for tracking active users on each document
5. Auto-save with debounce (save 2 sec after last keystroke)
6. Document stored as sequence of operations (not final text)
```
