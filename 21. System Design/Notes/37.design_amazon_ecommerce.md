# 37. Design: Amazon / E-Commerce Platform

## Step 1: Requirements

```
Functional:
  - Product catalog (search, browse, filter)
  - Shopping cart
  - Order placement and tracking
  - Payment processing
  - Inventory management
  - Reviews and ratings

Non-Functional:
  - High availability (99.99% - every minute of downtime = lost sales)
  - Low latency (product page loads < 200ms)
  - Handle traffic spikes (Black Friday, sales)
  - Strong consistency for inventory and payments
```

## Step 2: Microservices Architecture

```
┌─────────┐    ┌───────────┐    ┌─────────────────┐
│  Client │───→│API Gateway│───→│ Product Service  │ → Product DB + Elasticsearch
└─────────┘    │           │    ├─────────────────┤
               │           │───→│ Cart Service     │ → Redis (session-based cart)
               │           │    ├─────────────────┤
               │           │───→│ Order Service    │ → Order DB
               │           │    ├─────────────────┤
               │           │───→│ Payment Service  │ → Payment Gateway (Razorpay)
               │           │    ├─────────────────┤
               │           │───→│ Inventory Service│ → Inventory DB
               │           │    ├─────────────────┤
               │           │───→│ User Service     │ → User DB
               └───────────┘    └─────────────────┘
```

## Step 3: Order Placement Flow (Critical!)

```
User clicks "Place Order":

1. Cart Service → Get cart items
2. Inventory Service → RESERVE items (don't deduct yet!)
   "iPhone: 5 in stock → reserve 1 → 4 available"
   Reservation expires in 10 minutes (if payment fails, item goes back)

3. Payment Service → Process payment
   → Success: Continue
   → Failure: Release inventory reservation → show error

4. Order Service → Create order (status: "confirmed")

5. Inventory Service → DEDUCT reserved items (permanently)

6. Notification Service → Send order confirmation email/push

7. Return order details to user

This is a SAGA pattern! Each step has compensation:
  Payment fails → release inventory
  Order creation fails → refund payment + release inventory
```

## Step 4: Handling Flash Sales / Traffic Spikes

```
Problem: Black Friday → 100x normal traffic

Solutions:
1. Auto-scaling: More servers spin up automatically
2. Queue orders: Don't process all at once → queue → process steadily
3. Cache product pages: Same product page viewed by millions → cache it
4. Rate limiting: Prevent bots from buying all stock
5. Pre-warm CDN: Cache product images before sale starts

Inventory for flash sale:
  - Use Redis (fast, atomic operations)
  - DECR stock_iphone → atomic (no race condition)
  - If result < 0 → "Out of stock!" → reject order
```

## Step 5: Product Search

```
User searches "wireless headphones under 2000":

1. Query → Elasticsearch (full-text search)
2. Filters applied: price < 2000, category = "headphones"
3. Sort by relevance/price/rating
4. Results returned with: name, price, rating, thumbnail
5. Thumbnails served from CDN

Elasticsearch indexed fields:
  - Product name (full-text search)
  - Category
  - Price
  - Rating
  - Brand
  - Tags/keywords
```

## Key Design Decisions
```
1. Microservices (each domain = separate service)
2. Saga pattern for order processing (distributed transaction)
3. Redis for cart + inventory (fast, atomic operations)
4. Elasticsearch for product search
5. CDN for product images
6. Auto-scaling for traffic spikes
7. Strong consistency for payments + inventory
8. Eventual consistency for reviews, recommendations
```
