# 33. Design: Instagram / Twitter Feed

## Step 1: Requirements

```
Functional:
  - Post photos/tweets with caption
  - Follow/unfollow users
  - News feed (posts from people you follow, sorted by time/relevance)
  - Like, comment on posts

Non-Functional:
  - Feed loads in < 200ms
  - High availability (99.99%)
  - Eventual consistency OK (seeing a post 2 sec late is fine)

Scale:
  - 500M daily active users
  - Average user follows 200 people
  - Average user sees 500 posts/day
```

## Step 2: High-Level Design

```
POST Creation:
  User → API Gateway → Post Service → DB + S3 (media) + CDN
                                     → Fan-out Service (push to followers' feeds)

FEED Reading:
  User → API Gateway → Feed Service → Feed Cache (Redis) → return feed
```

```
┌──────┐    ┌───────────┐    ┌──────────────┐    ┌────────┐
│Client│───→│API Gateway│───→│ Post Service  │───→│  DB    │
└──────┘    └───────────┘    └──────┬───────┘    └────────┘
                                    │
                              ┌─────┴─────┐
                              │ Fan-out   │
                              │ Service   │
                              └─────┬─────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ↓               ↓               ↓
              Feed Cache      Feed Cache      Feed Cache
              (User A)        (User B)        (User C)
```

## Step 3: Feed Generation (THE Key Problem!)

### Approach 1: Fan-out on WRITE (Push Model)
```
When Ashish posts a photo:
  1. Save post in DB
  2. Get list of Ashish's followers: [Rahul, Priya, Amit, ... 1000 users]
  3. Push post to EACH follower's feed cache (Redis list)

  Rahul's feed cache: [Ashish's new post, older posts...]
  Priya's feed cache: [Ashish's new post, older posts...]

When Rahul opens Instagram:
  → Read from Rahul's pre-computed feed cache → INSTANT!

Pros: Feed reading is SUPER fast (already computed)
Cons: Celebrity posts → push to 10M followers = expensive & slow!
```

### Approach 2: Fan-out on READ (Pull Model)
```
When Rahul opens Instagram:
  1. Get list of people Rahul follows: [Ashish, Priya, Amit]
  2. Fetch recent posts from EACH person
  3. Merge and sort by time
  4. Return top 50 posts as feed

Pros: No extra work on posting
Cons: Feed loading is SLOW (querying many users each time)
```

### Approach 3: HYBRID (What Instagram/Twitter Actually Uses!)
```
Normal users (< 10K followers): Fan-out on WRITE
  → Pre-compute and push to followers' feed cache

Celebrities (> 10K followers): Fan-out on READ
  → Don't push to millions of feeds
  → When you open feed, pull celebrity posts separately and merge

Feed = Pre-computed cache + Real-time celebrity posts
```

## Step 4: Feed Ranking

```
Simple: Sort by timestamp (newest first)

Better (Algorithmic):
  Score = f(engagement, recency, relationship, content_type)

  Factors:
  - How recent is the post? (newer = higher score)
  - Does the user engage with this author often? (more interaction = higher)
  - Post engagement (lots of likes/comments = probably interesting)
  - Content type preference (user likes more photos than text)

  Machine Learning model predicts: "How likely is User X to engage with this post?"
  Higher probability = shown first
```

## Step 5: Storage

```
Posts:      Cassandra or DynamoDB (write-heavy, partitioned by user_id)
Media:     S3 + CDN (images/videos)
Feed:      Redis (pre-computed feed per user, list of post IDs)
Users:     PostgreSQL (relational data: followers, following)
Search:    Elasticsearch (search users, hashtags)
```

## Key Takeaway
```
The FEED is the hardest part of social media design.
Hybrid approach (push for normal users, pull for celebrities) is the answer.
Always mention this in interviews!
```
