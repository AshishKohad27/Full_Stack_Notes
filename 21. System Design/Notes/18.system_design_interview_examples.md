## Design 1: URL Shortener (like bit.ly)

### Requirements
```
- Convert long URL → short URL (e.g., bit.ly/abc123)
- Redirect short URL → original long URL
- Handle millions of URLs
- Short URL should be unique
```

### High-Level Design
```
Client → API Gateway → URL Service → Database
                                    → Cache (Redis)

POST /shorten { url: "https://very-long-url.com/page/123" }
→ Response: { short_url: "bit.ly/abc123" }

GET /abc123
→ Redirect to https://very-long-url.com/page/123
```

### Key Decisions
```
1. How to generate short URL?
   - Hash the long URL → take first 7 characters
   - Or use auto-increment ID → convert to base62 (a-z, A-Z, 0-9)
   - base62 with 7 chars = 62^7 = 3.5 trillion unique URLs!

2. Database
   - Table: id | short_url | long_url | created_at
   - NoSQL (DynamoDB) or SQL (PostgreSQL)

3. Cache
   - Cache popular URLs in Redis
   - 20% of URLs get 80% of traffic (Pareto principle)

4. Read vs Write
   - Read-heavy (1000 reads per 1 write)
   - Cache helps a lot!
```

## Design 2: Chat System (like WhatsApp)

### Requirements
```
- 1-to-1 chat and group chat
- Online/offline status
- Message delivery status (sent, delivered, read)
- Support millions of users
```

### High-Level Design
```
User A → WebSocket → Chat Server → Message Queue → Chat Server → WebSocket → User B

                                 → Database (store messages)
                                 → Cache (recent messages)
                                 → Push Notification Service (if offline)
```

### Key Decisions
```
1. Communication
   - WebSocket for real-time messaging
   - If user offline → store message → send via push notification

2. Message Storage
   - NoSQL (Cassandra) - good for write-heavy, time-series data
   - Partition by chat_id for efficient retrieval

3. Message delivery flow
   - User A sends message → Server stores in DB → marked "sent"
   - Server delivers to User B → marked "delivered"
   - User B opens chat → marked "read"

4. Group Chat
   - Message sent once → fan-out to all group members
   - Small group (< 100): fan-out on write (push to all)
   - Large group (> 100): fan-out on read (members pull)
```

## Design 3: Instagram/Twitter Feed

### Requirements
```
- Users post photos/tweets
- Users see a feed of posts from people they follow
- Feed should be personalized and ordered
- Handle millions of users
```

### High-Level Design
```
Post Creation:
User → API Gateway → Post Service → Database
                                   → Media Storage (S3)
                                   → CDN (serve images globally)

Feed Generation:
User → API Gateway → Feed Service → Cache (precomputed feed)
                                   → Database (if cache miss)
```

### Key Decisions
```
1. Feed Generation: Push vs Pull

   PUSH (Fan-out on Write):
   - When User A posts → immediately push to ALL followers' feeds
   - Pro: Feed is ready when follower opens app (fast read)
   - Con: Celebrity with 10M followers → 10M writes per post (expensive!)

   PULL (Fan-out on Read):
   - When User B opens feed → pull latest posts from all people they follow
   - Pro: No extra writes on posting
   - Con: Feed takes time to load (slow read)

   HYBRID (Best approach - used by Twitter):
   - Normal users: Push (fan-out on write)
   - Celebrities (>1M followers): Pull (fan-out on read)

2. Storage
   - Posts: NoSQL (Cassandra)
   - Media: Object Storage (AWS S3) + CDN
   - Feed Cache: Redis (precomputed feed per user)
```

## Design 4: Rate Limiter

### Requirements
```
- Limit API requests per user
- Different limits for different APIs
- Distributed (works across multiple servers)
```

### Design
```
Client → API Gateway → Rate Limiter (Redis) → Service
                            |
                     Check: user_id + api_endpoint
                     Count requests in current window
                     If under limit → ALLOW
                     If over limit → BLOCK (429)

Redis: fast, atomic operations, TTL support
Key:   rate_limit:user123:api_orders
Value: 45 (requests made)
TTL:   60 seconds (resets every minute)
```

## System Design Interview Template

```
Step 1: REQUIREMENTS (2-3 min)
  - Functional: What should system do?
  - Non-functional: Scale? Latency? Availability?
  - Ask questions! Don't assume.

Step 2: ESTIMATION (2-3 min)
  - How many users? DAU (Daily Active Users)?
  - How much data? Storage needed?
  - Read-heavy or write-heavy?

Step 3: HIGH-LEVEL DESIGN (5-10 min)
  - Draw boxes: Client, LB, Servers, DB, Cache
  - Show data flow with arrows
  - Identify APIs: POST /create, GET /feed

Step 4: DETAILED DESIGN (10-15 min)
  - Database schema
  - Which database? SQL or NoSQL?
  - Caching strategy
  - How to scale?

Step 5: BOTTLENECKS & TRADE-OFFS (5 min)
  - What if server crashes?
  - What if database is slow?
  - What if traffic spikes 10x?
  - CAP theorem trade-off?
```

## Quick Reference: Common Components

```
Need fast reads?          → Add Cache (Redis)
Need to handle millions?  → Horizontal scaling + Load Balancer
Need real-time?           → WebSocket
Need async processing?    → Message Queue (Kafka/RabbitMQ)
Need global speed?        → CDN
Need to store files?      → Object Storage (S3) + CDN
Need search?              → Elasticsearch
Need unique IDs?          → UUID / Snowflake ID generator
Need auth?                → API Gateway + JWT tokens
```
