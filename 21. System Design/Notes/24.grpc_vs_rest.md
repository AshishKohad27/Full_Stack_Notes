# 24. gRPC vs REST (Service-to-Service Communication)

- REST = how **client talks to server** (HTTP + JSON)
- gRPC = how **microservices talk to each other** (HTTP/2 + Protocol Buffers)

## REST API (What You Already Know)

```
Client → HTTP Request (JSON) → Server

POST /api/users
Content-Type: application/json
{
    "name": "Ashish",
    "age": 22
}

Response:
{
    "id": 1,
    "name": "Ashish",
    "age": 22,
    "message": "User created"
}

- Human-readable (JSON text)
- Works everywhere (browser, mobile, Postman)
- Simple to understand and debug
```

## gRPC (Google Remote Procedure Call)

```
Service A → gRPC call → Service B

Feels like calling a LOCAL function, but it runs on ANOTHER server!

// Service B has this function:
getUserById(123)  →  returns { name: "Ashish", age: 22 }

// Service A calls it as if it's local:
User user = userService.getUserById(123);

Behind the scenes: data sent over network in binary format (fast!)
```

## REST vs gRPC

| Feature       | REST                        | gRPC                        |
|---------------|-----------------------------|-----------------------------|
| Format        | JSON (text, human-readable) | Protocol Buffers (binary)   |
| Speed         | Slower (text parsing)       | 5-10x faster (binary)      |
| Protocol      | HTTP/1.1                    | HTTP/2                      |
| Streaming     | Limited                     | Full bidirectional streaming|
| Browser       | Works directly              | Needs proxy (grpc-web)      |
| Use case      | Client → Server             | Service → Service           |
| Learning      | Easy                        | More setup needed           |

## Real World Example

```
Uber (Microservices talking to each other):

  Ride Service ──gRPC──→ Pricing Service    (need speed!)
  Ride Service ──gRPC──→ Driver Service     (need speed!)
  Ride Service ──gRPC──→ Map Service        (need speed!)

  Mobile App ──REST──→ API Gateway          (needs browser compatibility)

Internal (between services): gRPC (fast!)
External (to clients): REST (compatible!)
```

## When to Use What?

```
Use REST:
  ✅ Client-facing APIs (browser, mobile)
  ✅ Public APIs (third-party developers)
  ✅ Simple CRUD operations
  ✅ When human readability matters

Use gRPC:
  ✅ Microservice-to-microservice communication
  ✅ Need high performance / low latency
  ✅ Real-time streaming (video, chat)
  ✅ Internal services that need speed
```

## Key Points
- REST = simple, compatible, JSON, for external/client APIs
- gRPC = fast, binary, for internal service-to-service calls
- Many companies use **BOTH**: REST for external, gRPC for internal
- gRPC is **5-10x faster** than REST due to binary serialization
- Netflix, Google, Uber use gRPC between their microservices
