# 6. Database Sharding & Replication

## A. Database Sharding (Split Data Across Databases)

- Sharding = **splitting one large database into smaller pieces** (shards)
- Each shard holds a **subset** of the data
- Like dividing a dictionary into Volume A-M and Volume N-Z

### How It Works

```
WITHOUT Sharding:                 WITH Sharding:
+--------------------+            +----------+  +----------+  +----------+
| ALL 100M Users     |            | Users    |  | Users    |  | Users    |
| in ONE database    |   ----->   | A - H    |  | I - P    |  | Q - Z    |
| SLOW!              |            | Shard 1  |  | Shard 2  |  | Shard 3  |
+--------------------+            +----------+  +----------+  +----------+
```

### Sharding Strategies

#### 1. Range-Based Sharding
```
Shard 1: Users with ID 1 - 1,000,000
Shard 2: Users with ID 1,000,001 - 2,000,000
Shard 3: Users with ID 2,000,001 - 3,000,000

Problem: Uneven distribution (Shard 1 might be busier if older users are more active)
```

#### 2. Hash-Based Sharding
```
shard_number = hash(user_id) % number_of_shards

User ID 101 → hash(101) % 3 = 2 → Shard 2
User ID 205 → hash(205) % 3 = 0 → Shard 0
User ID 333 → hash(333) % 3 = 1 → Shard 1

Better distribution! More evenly spread.
```

#### 3. Geographic Sharding
```
Shard India:  Users from India
Shard USA:    Users from USA
Shard Europe: Users from Europe

Data is closer to users = FASTER
```

### Sharding Challenges
```
1. JOINs across shards are HARD
   - User in Shard 1, Order in Shard 3 → can't do simple JOIN

2. Resharding is PAINFUL
   - Adding a new shard means redistributing data

3. Hotspot problem
   - One shard gets more traffic than others (celebrity users)

4. Cross-shard queries
   - "Give me ALL users sorted by name" → need to query ALL shards
```

## B. Database Replication (Copy Data for Safety)

- Replication = **keeping copies** of the same data on multiple servers
- If one database dies, another has the same data

### Master-Slave Replication (Most Common)

```
                    ┌──> Slave 1 (READ only)  ← reads go here
WRITES ──> Master ──┤
                    └──> Slave 2 (READ only)  ← reads go here

- Master: handles ALL writes (insert, update, delete)
- Slaves: handle ALL reads (select queries)
- Master copies data to Slaves automatically
```

### Why Master-Slave?
```
Most apps are READ-HEAVY:
  Instagram: 100 people VIEW a post, 1 person CREATES it

  Reads (90%) → distributed across many Slaves → FAST
  Writes (10%) → go to Master only
```

### Master-Master Replication
```
Master 1 ←──→ Master 2

- BOTH can read AND write
- Data syncs between them
- If one dies, other continues
- Risk: write conflicts (both update same data at same time)
```

## Replication vs Sharding

| Feature      | Replication                    | Sharding                      |
|--------------|--------------------------------|-------------------------------|
| What?        | Copy ALL data to multiple DBs  | Split data across multiple DBs|
| Purpose      | Availability + Read speed      | Handle more data + Write speed|
| Data         | Same data everywhere           | Different data on each shard  |
| If DB dies?  | Other copy takes over          | That shard's data unavailable |

## Real World Usage
```
Netflix:
  - Sharding: User data split by region
  - Replication: Each shard has replicas for backup

Instagram:
  - Sharding: Posts sharded by user ID
  - Replication: Master-Slave for read-heavy operations
```

## Key Points
- **Sharding** = split data → handle more data + writes
- **Replication** = copy data → safety + faster reads
- Most large systems use **BOTH** sharding AND replication
- Master-Slave: writes to master, reads from slaves
- Hash-based sharding gives best distribution
