# 32. Design: Chat System (WhatsApp/Messenger)

## Step 1: Requirements

```
Functional:
  - 1-to-1 chat
  - Group chat (up to 500 members)
  - Online/Offline status
  - Message status: Sent ✓, Delivered ✓✓, Read (blue ✓✓)
  - Push notifications (when offline)

Non-Functional:
  - Real-time (< 100ms latency)
  - Messages should never be lost
  - High availability
```

## Step 2: High-Level Design

```
                                     ┌──────────────┐
User A ──WebSocket──→ Chat Server 1 ─┤  Message     │
                                     │  Queue       │
User B ──WebSocket──→ Chat Server 2 ─┤  (Kafka)     │
                                     └──────┬───────┘
                                            │
                                     ┌──────┴───────┐
                                     │  Database    │
                                     │  (Cassandra) │
                                     └──────────────┘
```

## Step 3: Message Flow (1-to-1)

```
Ashish sends "Hello" to Rahul:

1. Ashish's phone → WebSocket → Chat Server 1
2. Chat Server 1 → stores message in DB (status: "sent" ✓)
3. Chat Server 1 → sends to Rahul's Chat Server via Message Queue

IF Rahul is ONLINE:
4. Chat Server 2 → WebSocket → Rahul's phone
5. Rahul's phone acknowledges → status: "delivered" ✓✓
6. Rahul opens chat → status: "read" (blue ✓✓)

IF Rahul is OFFLINE:
4. Message stored in DB
5. Push Notification Service → sends push notification to Rahul
6. When Rahul comes online → pull undelivered messages from DB
```

## Step 4: How to Know Which Server a User is Connected To?

```
Session Service (Redis):
  user:ashish → chat-server-1
  user:rahul  → chat-server-2
  user:priya  → chat-server-1

When Ashish sends message to Rahul:
  1. Look up Redis: "Rahul is on chat-server-2"
  2. Route message to chat-server-2
  3. Chat-server-2 pushes to Rahul via WebSocket
```

## Step 5: Group Chat

```
Group: "College Friends" (Ashish, Rahul, Priya)

Ashish sends "Hi everyone!":
  1. Message → Chat Server
  2. Server looks up group members: [Rahul, Priya]
  3. For each member:
     → Find their chat server (from Redis)
     → Route message to their server
     → Server pushes via WebSocket

Small groups (< 100): Fan-out on write (push to all immediately)
Large groups (> 100): Fan-out on read (members pull when they open group)
```

## Step 6: Database

```
Messages Table (Cassandra - good for write-heavy, time-series):
  Partition Key: chat_id (conversation ID)
  Clustering Key: timestamp (messages sorted by time)

  chat_id  | timestamp           | sender  | content  | status
  ─────────┼─────────────────────┼─────────┼──────────┼────────
  chat_1   | 2024-01-01 10:00:01 | ashish  | Hello    | read
  chat_1   | 2024-01-01 10:00:05 | rahul   | Hi!      | delivered
  chat_1   | 2024-01-01 10:00:10 | ashish  | How r u? | sent

Why Cassandra?
  - Write-heavy (billions of messages/day)
  - Time-series data (messages ordered by time)
  - Horizontally scalable
  - Partition by chat_id = all messages for a chat on same node
```

## Step 7: Online/Offline Status

```
Heartbeat approach:
  - Every user sends "I'm alive" signal every 5 seconds via WebSocket
  - Server updates: "ashish: last_seen = now" in Redis
  - If no heartbeat for 30 seconds → mark as offline

When Priya opens Ashish's chat:
  → Check Redis: ashish last_seen = 3 sec ago → "Online"
  → Check Redis: rahul last_seen = 10 min ago → "Last seen 10 min ago"
```

## Key Design Decisions

```
1. WebSocket for real-time messaging (not polling!)
2. Cassandra for message storage (write-heavy)
3. Redis for: session mapping, online status, caching
4. Kafka for: reliable message delivery between servers
5. Push Notification (FCM/APNs) for offline users
6. End-to-end encryption for security
```
