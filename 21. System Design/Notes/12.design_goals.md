# 12. Design Goals (The 5 Pillars)

- Every system you design should aim for these goals
- Like food should be tasty, healthy, and affordable — systems should be scalable, available, consistent

---

## 1. Scalability

### What?
- System's ability to handle **growing** amount of work
- 1000 users today → 1 million users tomorrow → system should still work

### Real World Example
```
Local Bakery:
  - 1 cashier handles 10 customers/hour
  - Bakery becomes popular → 100 customers/hour
  - Solution: Open 5 more counters (SCALE!)

Same with systems:
  - 1 server handles 1000 requests/sec
  - App goes viral → 100,000 requests/sec
  - Solution: Add more servers + load balancer
```

### Types
```
Vertical Scaling = Bigger machine (more RAM, CPU)
Horizontal Scaling = More machines (add servers)
```

## 2. Availability

### What?
- How much time system is **up and running**
- Measured in "nines"

### The Nines Table

| Availability | Downtime per Year     | Called         |
|--------------|----------------------|----------------|
| 99%          | 3.65 days            | Two nines      |
| 99.9%        | 8.76 hours           | Three nines    |
| 99.99%       | 52.6 minutes         | Four nines     |
| 99.999%      | 5.26 minutes         | Five nines     |
| 99.9999%     | 31.5 seconds         | Six nines      |

### How to Achieve High Availability?
```
1. Redundancy - Multiple servers (if one dies, others work)
2. Load Balancer - Distribute traffic, skip dead servers
3. Health Checks - Monitor servers, restart if needed
4. Failover - Automatic switch to backup
5. Multiple Data Centers - If entire building goes down, another takes over
```

### Real World
```
Google: 99.999% availability (5 nines)
  = only 5 minutes downtime per YEAR!

Banks: 99.99% availability (4 nines)
  = only 52 minutes downtime per year
```

## 3. Consistency

### What?
- All users see the **SAME data at the SAME time**
- When data is updated, everyone sees the update

### Two Types

#### Strong Consistency
```
WRITE: Update balance to Rs.5000
READ (immediately after): Rs.5000 ✅ (always correct)

Every read returns the MOST RECENT write.
Like a bank: withdraw Rs.1000 → balance IMMEDIATELY shows Rs.4000

Used in: Banking, stock trading, booking systems
```

#### Eventual Consistency
```
WRITE: Update profile picture
READ (immediately after): Old picture 😕 (might be stale for a moment)
READ (after some time): New picture ✅

Data WILL become consistent, just not immediately.
Like Instagram: you change your pic, friend in another country
might see old pic for a few seconds.

Used in: Social media, DNS, search engines
```

### Strong vs Eventual

| Feature           | Strong Consistency       | Eventual Consistency      |
|-------------------|--------------------------|---------------------------|
| Speed             | Slower                   | Faster                    |
| Accuracy          | Always up-to-date        | Might be stale briefly    |
| Use case          | Banking, payments        | Social media, analytics   |
| Implementation    | Harder                   | Easier                    |
| Availability      | Lower (waits for sync)   | Higher (serves instantly) |

## 4. No Single Point of Failure (SPOF)

### What?
- SPOF = a component that if it fails, the **entire system goes down**
- Goal: NO component should be a single point of failure

### Example
```
BAD (has SPOF):
  User → [ONE Server] → [ONE Database]
  If server dies → ENTIRE app is down!
  If database dies → ENTIRE app is down!

GOOD (no SPOF):
  User → [Load Balancer] → [Server 1] → [DB Master]
                          → [Server 2] → [DB Slave 1]
                          → [Server 3] → [DB Slave 2]
  If Server 2 dies → Server 1 and 3 still work!
  If DB Master dies → Slave promoted to Master!
```

### How to Eliminate SPOF?
```
1. Multiple servers behind load balancer
2. Database replication (master + slaves)
3. Multiple data centers in different locations
4. Redundant load balancers (even LB can be SPOF!)
5. DNS failover
```

## 5. Fault Tolerance

### What?
- System **continues to work** even when some parts FAIL
- Closely related to "No SPOF" but broader

### Example
```
Netflix:
  - Data Center A goes down (earthquake, fire)
  - Automatically switches to Data Center B
  - Users don't even notice!
  - Netflix is fault tolerant

Amazon:
  - One payment server crashes
  - Load balancer routes to other payment servers
  - User's checkout continues without error
```

### How to Achieve?
```
1. REDUNDANCY - Have backups of everything
2. REPLICATION - Copy data across servers/regions
3. HEALTH CHECKS - Detect failures quickly
4. AUTO-FAILOVER - Automatically switch to backup
5. GRACEFUL DEGRADATION - Show limited features instead of crashing
   Example: Netflix can't load recommendations → still plays videos
```

## CAP Theorem -- VERY IMPORTANT FOR INTERVIEWS!

### What?
- In a **distributed system**, you can only guarantee **2 out of 3**:

```
         C (Consistency)
        / \
       /   \
      /     \
     A ----- P
(Availability) (Partition Tolerance)
```

| Letter | Meaning                                                    |
|--------|------------------------------------------------------------|
| C      | Consistency - all nodes see same data at same time         |
| A      | Availability - every request gets a response               |
| P      | Partition Tolerance - system works even if network splits  |

### You MUST choose P (network failures WILL happen)
### So your real choice is: CP or AP

```
CP (Consistency + Partition Tolerance):
  - When network splits, system blocks until consistent
  - Use for: Banking, payments
  - Examples: MongoDB, HBase, Redis

AP (Availability + Partition Tolerance):
  - When network splits, system stays available but may serve stale data
  - Use for: Social media, shopping carts
  - Examples: Cassandra, DynamoDB, CouchDB
```

### Interview Answer
```
"You can't have all three in a distributed system.
Since network partitions are unavoidable, we choose between:
  - CP: Block requests until data is consistent (banking)
  - AP: Serve requests even with stale data (social media)"
```

|-------------------|----------------------------------|-----------------------------|
| Scalability       | Handle more load                 | Horizontal scaling, sharding|
