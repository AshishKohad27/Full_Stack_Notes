# 11. API Gateway

- API Gateway = **single entry point** for ALL client requests
- Sits between Client and Microservices
- Like a **reception desk** at a company - you don't go directly to departments

## Why API Gateway?

```
WITHOUT API Gateway:                  WITH API Gateway:
Client must know every service URL    Client knows ONE URL

Client → auth-service.com/login       Client → api.myapp.com/login
Client → order-service.com/orders            → api.myapp.com/orders
Client → payment-service.com/pay            → api.myapp.com/pay
Client → notify-service.com/send            → api.myapp.com/notify

Client manages 10 URLs = COMPLEX      Client manages 1 URL = SIMPLE
```

## How It Works

```
                                    ┌── Auth Service
Client ──> API Gateway ──┼── Order Service
           (Single Door)          ├── Payment Service
                                    └── Notification Service

Gateway receives request → routes to correct service → returns response
```

## What API Gateway Does

### 1. Routing
```
/api/users/*    → routes to User Service
/api/orders/*   → routes to Order Service
/api/payments/* → routes to Payment Service
```

### 2. Authentication & Authorization
```
Every request passes through gateway first:

Client → API Gateway → "Is this user logged in?" → "Does user have permission?"
                            ↓ YES                         ↓ YES
                      Forward to service              Return data
                            ↓ NO                          ↓ NO
                      Return 401 Unauthorized        Return 403 Forbidden

Services don't need to handle auth themselves!
```

### 3. Rate Limiting
```
"User can make max 100 requests per minute"

Request 1-100:  ✅ Allowed
Request 101:    ❌ 429 Too Many Requests

Prevents abuse and protects your servers.
```

### 4. Request/Response Transformation
```
Mobile app needs small response:
  Gateway removes unnecessary fields before sending

Web app needs detailed response:
  Gateway sends full response

One service, different response formats per client!
```

### 5. Load Balancing
```
Gateway can also balance load across multiple instances of same service:

/api/orders → Order Service Instance 1
            → Order Service Instance 2
            → Order Service Instance 3
```

### 6. Caching
```
Same request asked 100 times?
Gateway caches the response, doesn't hit service every time.
```

### 7. Logging & Monitoring
```
ALL requests pass through gateway → perfect place to:
  - Log every request
  - Monitor response times
  - Track errors
  - Collect analytics
```

## API Gateway vs Load Balancer

| Feature          | API Gateway                  | Load Balancer              |
|------------------|------------------------------|----------------------------|
| Purpose          | Route to correct SERVICE     | Distribute to same service |
| Intelligence     | URL, headers, auth, transform| IP, port, health           |
| Layer            | Layer 7 (application)        | Layer 4 or 7               |
| Features         | Auth, rate limit, transform  | Health check, distribute   |
| Example          | /users → User Service        | → Server 1, 2, or 3       |

```
In real systems, both work together:

Client → API Gateway → Load Balancer → Service Instance 1
                                     → Service Instance 2
                                     → Service Instance 3
```

## Key Points
- API Gateway = single entry point for all requests
- Handles: routing, auth, rate limiting, caching, monitoring
- Clients talk to ONE URL, gateway routes to correct service
- Essential for **microservices** architecture
- Also provides **security layer** (auth check before reaching services)
