# 3. Database Basics

- Database = organized place to **store and retrieve data**
- Without database, data is lost when server restarts

## Two Types: SQL vs NoSQL

### SQL (Relational Database)
- Data stored in **tables** (rows and columns) - like Excel
- Has strict **schema** (structure defined in advance)
- Uses **SQL language** to query
- Examples: MySQL, PostgreSQL, Oracle, SQL Server

```
Users Table:
+----+--------+-----+------------------+
| id | name   | age | email            |
+----+--------+-----+------------------+
| 1  | Ashish | 22  | ashish@mail.com  |
| 2  | Rahul  | 25  | rahul@mail.com   |
+----+--------+-----+------------------+
```

### NoSQL (Non-Relational Database)
- Data stored as **documents, key-value, graphs** etc.
- **Flexible schema** (no fixed structure)
- Examples: MongoDB, Redis, Cassandra, DynamoDB

```
// MongoDB Document (JSON-like)
{
    "id": 1,
    "name": "Ashish",
    "age": 22,
    "hobbies": ["coding", "gaming"],   // arrays allowed!
    "address": {                        // nested objects allowed!
        "city": "Mumbai",
        "pin": 400001
    }
}
```

## SQL vs NoSQL Comparison

| Feature        | SQL                        | NoSQL                        |
|----------------|----------------------------|------------------------------|
| Structure      | Tables (rows/columns)      | Documents, key-value, graph  |
| Schema         | Fixed (strict)             | Flexible (dynamic)           |
| Scaling        | Vertical (bigger server)   | Horizontal (more servers)    |
| Relationships  | Strong (JOINs)             | Weak (denormalized)          |
| Best for       | Complex queries, banking   | Big data, real-time apps     |
| Examples       | MySQL, PostgreSQL          | MongoDB, Redis, Cassandra    |
| ACID           | Yes                        | Some (depends on DB)         |

## When to Use What?

```
Use SQL when:
  - Data has clear relationships (users, orders, products)
  - You need complex queries with JOINs
  - Data integrity is critical (banking, healthcare)
  - Structure is well-defined and won't change much

Use NoSQL when:
  - Data structure changes frequently
  - You need to scale horizontally (millions of users)
  - Data is unstructured (social media posts, logs)
  - Speed is priority over strict consistency
```

## ACID Properties (SQL)

```
A - Atomicity:    Transaction is ALL or NOTHING
                  (Transfer money: debit AND credit both happen, or neither)

C - Consistency:  Database goes from one valid state to another
                  (Balance can't be negative if rule says so)

I - Isolation:    Transactions don't interfere with each other
                  (Two people withdrawing at same time don't cause issues)

D - Durability:   Once committed, data survives crashes
                  (Power goes off? Your data is still saved)
```

## Real World Examples

```
Instagram:
  - User profiles, followers --> SQL (PostgreSQL)
  - Posts, stories, media    --> NoSQL (Cassandra)
  - Cache                   --> NoSQL (Redis)

Amazon:
  - Products, orders         --> SQL
  - Product recommendations  --> NoSQL (DynamoDB)
  - Search                   --> Elasticsearch (NoSQL)
```

## Key Points
- SQL = structured, strict, relationships, ACID
- NoSQL = flexible, scalable, fast, eventually consistent
- Most real apps use **BOTH** (SQL + NoSQL) together
- Choose based on your data and use case
